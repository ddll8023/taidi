# 任务三结果工作台设计说明

## 1. 背景与目标

基于赛题文档 [doc/B题-上市公司财报-智能问数助手.md](doc/B题-上市公司财报-智能问数助手.md) 与现有任务二页面 [frontend/src/views/task2/Task2WorkbenchView.vue](frontend/src/views/task2/Task2WorkbenchView.vue)，新增一个“任务三结果工作台”页面，用于承接任务三问题导入、单题/批量执行、结果查看与 `result_3.xlsx` 导出。

设计目标如下：

1. 页面交互风格与任务二保持一致，降低学习成本。
2. 结果展示严格贴合任务三题目要求，支持 `content`、`image`、`references` 三类输出。
3. 在不破坏任务二工作台简洁性的前提下，额外展示任务三特有的执行过程信息，包括执行计划、校验结果、知识库检索摘要等。
4. 参考文献不在主页面铺开，而是通过整题统一按钮进入弹窗集中查看。

## 2. 页面定位

任务三结果工作台是任务三模块下的第二个核心页面，与现有知识库管理页并列存在：

- 知识库管理：面向附件 5 知识库的初始化、上传、切块、向量化与状态管理。
- 结果工作台：面向附件 6 题目导入、执行、查看、导出。

因此，任务三模块下建议同时保留：

- `/task3/knowledge-base`
- `/task3/workbench`

导航中新增“任务三结果工作台”入口，不替换现有知识库管理入口。

## 3. 设计原则

### 3.1 复用任务二骨架

整体沿用任务二的双栏工作台结构：

- 顶部工具栏
- 左侧题目列表
- 右侧题目详情
- 右侧未选择题目时的空状态

其中需要尽量贴近 [frontend/src/views/task2/Task2WorkbenchView.vue](frontend/src/views/task2/Task2WorkbenchView.vue) 已有风格，包括：

- 顶部工具栏按钮层级与摆放顺序
- 左侧状态筛选下拉
- 左侧题目卡片选中态 / 未选中态
- 右侧卡片化详情区块顺序
- 空状态的图标、标题、说明文案组织方式

这样可以最大化复用用户认知和前端实现模式。

### 3.2 结果结构服从赛题格式

页面展示可以优化层次，但底层结果数据结构应与任务三要求一致：

```json
[
  {
    "Q": "问题",
    "A": {
      "content": "文字回答",
      "image": ["./result/B2001_1.jpg"],
      "references": [
        {
          "paper_path": "研报路径",
          "text": "原文摘要",
          "paper_image": "图表路径"
        }
      ]
    }
  }
]
```

页面只是二次组织，不改变结果原始语义。

### 3.3 任务三特性增强但不过度复杂

任务三需要比任务二多展示内部过程信息，但页面不做多标签页、多层工作流拆分，仍然坚持单页详情滚动查看，避免把工作台做成新的流程系统。

## 4. 页面整体结构

整体采用与任务二一致的两栏布局。

### 4.1 顶部工具栏

顶部工具栏建议保留以下元素：

#### 左侧
- 页面标题：任务三结果工作台
- 工作区统计信息：
  - 题目总数
  - 已完成
  - 待处理
  - 失败

#### 右侧操作按钮
- 上传附件6
- 批量回答未完成
- 批量回答全部
- 导出结果
- 刷新

### 4.2 左侧题目列表

列表风格与任务二保持一致，每条题目展示：

- `question_code`
- `question_type`
- `status`

并保留与任务二一致的状态筛选能力，至少支持：

- 全部
- 待处理
- 已完成
- 失败

不额外堆叠执行计划或 references 数量，避免信息密度过高。

### 4.3 右侧空状态

当未选中题目时，右侧保持与任务二一致的空状态布局：

- 居中图标
- 页面标题
- 简短说明文案

说明文案需明确这是“任务三结果工作台”，并说明支持上传附件6、回答题目、导出 `result_3.xlsx`。

### 4.4 右侧题目详情区

右侧为任务三重点展示区域，在任务二结构基础上扩展。

## 5. 详情区信息架构

### 5.1 头部信息区

展示：

- 题号
- 题型
- 当前状态（可选）
- 操作按钮：
  - 回答本题
  - 重新回答
  - 删除回答
  - 查看参考文献

其中“查看参考文献”为任务三新增的统一入口按钮，展示整道题的全部 references 汇总，而非每轮问答单独按钮。

### 5.2 原始问题

沿用任务二展示方式，使用 `pre` 区块展示原始 JSON 问题内容，用于对照附件 6 原题格式。

### 5.3 回答结果

沿用任务二的问答卡片结构：

- 遍历 `answer_json`
- 每轮展示 `Q`
- `A.content` 使用 markdown 渲染
- `A.image` 渲染图片列表

这里不直接展开 `references`，避免详情区过长。

需要明确两类空态：

- 若题目尚未回答完成，则显示“暂无回答结果”或仅展示处理中状态
- 若某轮回答没有图片，则不渲染图片区块

图片路径需要遵循与任务二一致的前端访问映射策略。若后端仍返回类似 `./result/xxx.jpg` 的相对路径，则前端必须统一转换成可访问地址后再展示，避免直接暴露原始相对路径。

### 5.4 执行计划

新增任务三专属区块，用于展示规划驱动执行链路的关键信息。

建议展示以下字段：

- 计划摘要
- 步骤总数
- 各步骤列表
  - `step_id`
  - `step_type`
  - `goal`
  - `status`
  - `depends_on`（如有）
  - 简要输出摘要（如有）

展示目标是帮助用户理解任务三“先规划、再执行”的工作过程，但不要求完整回放所有中间上下文。

### 5.5 校验结果

新增任务三专属区块，用于展示任务三可靠性增强模块的输出。

建议展示内容：

- `passed`
- `errors`
- `warnings`
- 核心统计摘要，例如：
  - 完整性校验状态
  - 一致性校验状态
  - 合理性校验状态
  - 引用有效性校验状态

视觉上建议通过成功 / 警告 / 失败的色块或 badge 区分。

### 5.6 知识库检索摘要

新增任务三专属区块，用于说明本题是否触发知识库检索，以及命中概况。

建议展示：

- 是否触发检索
- 命中文档数量
- 命中文档类型（个股研报 / 行业研报 / 财报）
- 命中股票代码或过滤条件摘要
- 是否生成 references

这里只给摘要，不直接铺开原文证据。

### 5.7 SQL 语句

保留任务二的 SQL 展示形式，单独以深色代码块展示。

若当前题目没有 SQL，或当前执行链路并未生成 SQL，则该区块隐藏，不显示空白容器。

### 5.8 图表类型

保留任务二的图表类型展示逻辑，用 badge 或胶囊标签展示图表类型。

若当前题目无图表，则隐藏该区块。

### 5.9 错误信息

保留任务二的错误区块，用于展示执行失败、导出失败或中间步骤异常信息。

失败题需优先保证错误信息可见，避免用户只能看到空白结果区。

### 5.10 时间信息

保留任务二的底部时间字段：

- 创建时间
- 回答时间

## 6. 参考文献弹窗设计

### 6.1 触发方式

由详情头部统一按钮“查看参考文献”打开。

不是每轮问答各自一个按钮，而是整题级统一入口。

### 6.2 汇总策略

弹窗中展示当前题目全部轮次 `A.references` 的汇总结果。

建议汇总逻辑：

1. 遍历 `answer_json` 下每轮回答的 `A.references`
2. 拉平为一个 references 列表
3. 按 `paper_path + text + paper_image` 去重
4. 统一展示

### 6.3 弹窗内容结构

每条 reference 展示：

- `paper_path`
- `text`
- `paper_image`

若 `paper_image` 是图片路径，则显示图片预览或可点击链接；若仅为图表名称，则按文本展示。

若没有 references，则显示空状态：

- 当前题目暂无参考文献

### 6.4 组件建议

建议封装独立组件：

- `frontend/src/components/task3/Task3ReferencesModal.vue`

这样可避免结果页文件过重，也便于后续复用。

## 7. 首版展示范围控制

虽然任务三可以展示执行计划、校验结果、知识库检索摘要等扩展信息，但首版实现应优先保证“任务二同构工作台”主流程落地。

首版必做范围：

- 导入附件6
- 题目列表
- 题目详情
- 单题回答
- 批量回答
- 导出 `result_3.xlsx`
- 统一 references 弹窗

首版增强范围：

- 执行计划
- 校验结果
- 知识库检索摘要

若后端当前尚未稳定返回这些增强数据，则前端应采用“有数据再展示”的策略：

- 有数据：显示对应区块
- 无数据：隐藏区块，不占位

避免为了展示扩展能力而反向要求页面出现大量空白模块。

## 8. 路由与导航建议

### 8.1 路由

在 [frontend/src/router/index.js](frontend/src/router/index.js) 的 `/task3` 下新增子路由：

- `path: 'workbench'`
- `name: 'Task3Workbench'`
- `component: () => import('@/views/task3/Task3WorkbenchView.vue')`

不移除现有知识库管理页。

### 8.2 导航

在 [frontend/src/data/navigation.js](frontend/src/data/navigation.js) 的“智能助手”分组下新增：

- 标题：任务三结果工作台
- 描述：上传附件6、批量回答问题、导出 result_3.xlsx
- 图标：建议沿用 `list-check` 或使用更贴合工作台语义的图标

## 9. Store 设计建议

参考 [frontend/src/stores/task2.js](frontend/src/stores/task2.js) 新增任务三专用 store：

- `frontend/src/stores/task3.js`

建议状态结构：

- `workspace`
- `questions`
- `currentQuestion`
- `isLoading`
- `isUploading`
- `isProcessing`
- `isExporting`
- `error`

任务三扩展状态建议增加：

- `showReferencesModal`
- `aggregatedReferences`

可选派生数据：

- `currentPlanSummary`
- `currentVerificationSummary`
- `currentRetrievalSummary`

其中扩展摘要优先由 `currentQuestion` 计算得出，避免重复存储。

## 10. API 对接建议

当前前端已有基础任务三接口 [frontend/src/api/task3.js](frontend/src/api/task3.js)，包括：

- `processQuestion`
- `createPlan`
- `executePlan`
- `exportResult`
- `exportSingleResult`
- `verifyResult`

这意味着：当前任务三前端更接近“单次问答/计划执行型接口”，还不是任务二那种完整工作台接口集合。

因此设计上需要明确区分：

### 10.1 当前已存在能力
- 单次问题处理
- 执行计划生成
- 执行计划执行
- 单题导出
- 结果校验

### 10.2 为任务三工作台补齐的能力
若要完整对标任务二工作台流程，后续需要新增或补齐“工作台型”接口语义，例如：

- 工作区信息获取
- 附件6导入
- 题目列表获取
- 单题详情获取
- 单题执行
- 单题删除
- 单题重跑
- 批量执行
- 最新导出信息获取

若后端尚未提供这些接口，则需要在后续实现阶段一并补齐，而不能在前端设计文档中默认这些能力已经存在。

同时，题目切换时建议优先采用“按题目 ID 拉取详情”的模式，与任务二 store 的工作方式保持一致，避免在列表接口中塞入完整详情导致列表负载过大。

## 12. 结果一致性要求

任务三页面的结果展示必须满足题目要求：

1. 最终导出目标是 `result_3.xlsx`
2. 页面展示的回答结构与导出结构一致
3. `references` 中必须保留：
   - `paper_path`
   - `text`
   - `paper_image`
4. 图片路径展示应与当前项目图片访问机制兼容

## 13. 推荐落地方案

采用“任务二同构增强版”：

- 页面骨架与任务二尽量一致
- 操作模式与任务二一致
- 结果展示结构与任务二相近
- 任务三新增信息以详情区新增区块 + 统一 references 弹窗的方式补充

这是当前实现成本最低、体验一致性最好、最符合赛题要求的一种方案。

## 14. 后续实现顺序建议

建议后续按以下顺序实现：

1. 新增路由与导航入口
2. 新建任务三 store 和工作台页面骨架
3. 先实现和任务二同构的基础能力：导入、列表、详情、单题执行、批量执行、导出
4. 再补任务三扩展展示：执行计划、校验结果、知识库检索摘要
5. 最后实现 references 汇总弹窗与细节打磨

## 15. 结论

任务三结果工作台应作为一个独立页面存在，保持任务二工作台的布局与交互习惯，同时补充任务三专属的过程信息与 references 弹窗能力。页面展示以赛题要求的结果结构为准，避免前端为展示方便而破坏导出结构或数据语义。
