## 一、设计背景与目标

### 1.1 问题定义

任务二要求构建面向上市公司财报的"智能问数"助手，实现自然语言输入到结构化查询的全流程自动化。具体目标包括：

1. **自然语言解析**：用户以自然语言提问，系统自动理解意图并生成SQL查询
2. **多轮对话支持**：能够理解上下文语境，处理追问和跟进式查询
3. **意图澄清机制**：当查询意图模糊时，主动向用户澄清核心要素
4. **可视化图表生成**：根据查询结果自动选择合适的图表类型并生成分析结论
5. **派生指标计算**：支持同比增长、环比增长、复合增长率等复杂指标计算
6. **零技术门槛**：用户无需了解SQL、数据库结构，直接用自然语言交互

### 1.2 技术挑战

**挑战1：意图理解的准确性**
- 自然语言表达多样化，同一问题可能有不同表述
- 需要准确识别公司、指标、时间三个核心槽位
- 需要区分不同查询类型（单值查询、趋势分析、排名对比）
- 需要识别派生指标关键词（同比、环比、复合增长率等）

**挑战2：SQL生成的安全性**
- LLM可能生成不安全的SQL（DDL/DML语句）
- LLM可能引用不存在的表或字段
- 需要严格的安全校验机制防止恶意操作

**挑战3：多轮对话上下文管理**
- 需要记忆前一轮对话的槽位信息
- 需要处理指代消解（"它"、"该公司"、"其中"等）
- 需要合理更新上下文状态
- 需要支持集合指代（"筛选出的公司"、"这些企业"等）

**挑战4：意图澄清的触发时机**
- 如何判断信息是否充分？不同查询类型的必需槽位不同
- 如何生成自然的澄清问题？避免机械化回复
- 如何引导用户补全信息？提供友好的交互体验

**挑战5：可视化图表的智能选择**
- 不同查询类型适合不同图表（折线图、柱状图、雷达图等）
- 需要根据数据特征自动判断最佳图表类型
- 需要生成高质量的分析结论，不是简单罗列数字

**挑战6：派生指标的计算**
- 如何识别用户需要计算派生指标？
- 如何生成复杂的SQL查询（窗口函数、聚合函数等）
- 如何解释计算结果和计算逻辑？

**挑战7：不支持查询的友好提示**
- 如何识别数据源不支持的问题？
- 如何主动告知用户不支持原因？
- 如何提供替代方案建议？

**关键设计（与实际代码完全一致）**：
- **context_slots字段**：JSON类型，存储完整的槽位快照，包括新增的capability、derived_metric_type、last_result_companies
- **intent_result字段**：JSON类型，记录每条消息的意图解析结果，便于追溯
- **chart_paths字段**：JSON数组，存储图表路径（API URL格式），支持图表持久化和删除清理

### 3.2 多轮对话状态管理机制

**核心设计**：通过会话表的context_slots字段实现槽位快照持久化。

**状态流转示例**：

```
第一轮对话：
用户："金花股份利润总额是多少"
IntentResult: {
    company: {value: "金花股份", type: "stock_abbr"},
    metric: {field: "total_profit", table: "income_sheet", display_name: "利润总额"},
    time_range: null,  // 缺失
    query_type: single_value,
    capability: DIRECT_FIELD,
    derived_metric_type: null
}
→ 检测缺失time_range
→ 澄清："请问您想查询哪个报告期的数据？"
→ 更新context_slots（保存已有槽位）

第二轮对话：
用户："2025年第三季度的"
IntentResult: {
    company: null,  // 未提及
    metric: null,  // 未提及
    time_range: {report_year: 2025, report_period: "Q3"},
    query_type: single_value
}
→ 合并上下文：
   - company继承历史值 → 金花股份
   - metric继承历史值 → 利润总额
   - time_range使用新值 → 2025Q3
→ 槽位完整，生成SQL
→ 查询结果："金花股份2025年第三季度利润总额为3140万元"
→ 更新context_slots（完整槽位）

第三轮对话：
用户："它的净利润是多少"
→ 指代消解："它" → "金花股份"
→ 解析意图：company=金花股份, metric=净利润, time_range=null
→ 合并上下文：time_range继承历史值 → 2025Q3
→ 槽位完整，直接查询

第四轮对话（集合指代）：
用户："2024年净利润最高的top5企业"
→ 查询返回5家公司
→ 保存last_result_companies: [
    {value: "平安银行", type: "stock_abbr"},
    {value: "招商银行", type: "stock_abbr"},
    ...
]

第五轮对话：
用户："其中净利润年同比最高的是哪家"
→ 指代消解："其中" → "平安银行、招商银行、...中"
→ 查询范围限定在5家公司内
```

**合并规则（chat.py:1216-1293）**：
1. **新槽位优先**：用户明确提及的槽位覆盖历史值
2. **未明确继承**：用户未提及的槽位继承历史值
3. **缺失保持缺失**：历史槽位也为缺失时，保持缺失状态
4. **特殊字段继承**：capability、derived_metric_type、last_result_companies等也继承

## 四、意图识别实现机制

### 4.1 意图解析流程

意图识别通过LLM调用实现，核心流程包括：

```
步骤1：构建Schema信息
• 提取SCHEMA_INFO配置（表名、字段名、中文描述）
• 格式化为文本格式，便于LLM理解
• 示例：
  表名: income_sheet
    身份字段: report_id, stock_code, stock_abbr...
    指标字段:
      net_profit: 净利润(万元)
      total_profit: 利润总额(万元)
```

### 5.3 SQL安全校验机制

SQL校验分为四层检查（chat.py:1078-1098）：

```
第一层：禁止关键字检查
• 检查DDL/DML关键字：DROP、DELETE、UPDATE、INSERT、ALTER、CREATE、TRUNCATE、GRANT、REVOKE
• 正则匹配：\b{keyword}\b（单词边界匹配）
• 失败返回：SQL包含禁止关键字: {keyword}

第二层：起始语句检查
• 必须以SELECT开头
• 防止执行其他类型语句
• 失败返回：SQL必须以SELECT开头

第三层：表名白名单检查
• 提取FROM/JOIN后的表名
• 检查表名是否在ALLOWED_TABLES列表中
• 失败返回：SQL引用了不允许的表: {table}

第四层：自动添加LIMIT
• 若SQL不包含LIMIT，自动添加LIMIT 1000
• 防止查询过大数据集
• 优化数据库性能
```

**校验流程代码**：

```python
def validate_sql(sql: str) -> tuple[bool, str]:
    # 第一层：禁止关键字
    for keyword in FORBIDDEN_KEYWORDS:
        pattern = r"\b" + keyword + r"\b"
        if re.search(pattern, sql.upper()):
            return False, f"SQL包含禁止关键字: {keyword}"
    
    # 第二层：起始语句
    if not sql.upper().startswith("SELECT"):
        return False, "SQL必须以SELECT开头"
    
    # 第三层：表名白名单
    tables = re.findall(r"\b(?:FROM|JOIN)\s+(\w+)", sql.upper())
    for table in tables:
        if table.lower() not in [t.lower() for t in ALLOWED_TABLES]:
            return False, f"SQL引用了不允许的表: {table}"
    
    # 第四层：自动LIMIT
    if "LIMIT" not in sql.upper():
        sql = sql.rstrip(";") + " LIMIT 1000"
    
    return True, ""
```

**SQL模板结构**：
```sql
WITH qualified_periods AS (
    -- 步骤1：筛选满足条件的记录，并分配时间序号
    SELECT 
        stock_code, stock_abbr, report_year, report_period, {metric_field},
        ROW_NUMBER() OVER (
            PARTITION BY stock_code 
            ORDER BY report_year, 
                CASE report_period 
                    WHEN 'Q1' THEN 1 
                    WHEN 'HY' THEN 2 
                    WHEN 'Q3' THEN 3 
                    WHEN 'FY' THEN 4 
                END
        ) as rn
    FROM {table_name}
    WHERE {condition}
      AND report_year BETWEEN {start_year} AND {end_year}
),
company_continuous_count AS (
    -- 步骤2：按公司统计满足条件的期数
    SELECT stock_code, stock_abbr
    FROM qualified_periods
    GROUP BY stock_code, stock_abbr
    HAVING COUNT(*) >= {period_count}
)
-- 步骤3：返回满足连续性要求的公司详细数据
SELECT q.stock_code, q.stock_abbr, q.report_year, q.report_period, q.{metric_field}
FROM qualified_periods q
INNER JOIN company_continuous_count c ON q.stock_code = c.stock_code
ORDER BY q.stock_code, q.report_year
```

**技术要点**：
- 使用ROW_NUMBER()窗口函数为每个公司的报告期分配连续序号
- 使用CASE语句将报告期转换为可排序的数字（Q1=1, HY=2, Q3=3, FY=4）
- 按公司分组统计满足条件的期数，筛选出>=N的公司
- 返回这些公司在指定时间范围内的所有满足条件的记录

```python
def _detect_business_definition_needed(question: str) -> dict | None:
    """检测问题中是否包含需要业务定义澄清的关键词"""
    config = _load_derived_metrics_config()

                }
```

## 九、意图澄清机制设计

### 9.1 澄清触发条件

根据查询类型动态判断必需槽位：

| 查询类型 | 必需槽位 | 缺失触发澄清 |
|---------|---------|-------------|
| single_value | company, metric, time_range | 任一缺失触发 |
| trend | company, metric, time_range_for_trend | 任一缺失触发 |
| ranking | metric, time_range | 任一缺失触发（company可选） |
| comparison | metric, time_range | 任一缺失触发（company可选） |
| continuity | metric, time_range, period_count | 任一缺失触发 |

**检测逻辑（chat.py:937-974）**：

```python
def check_missing_slots(intent: IntentResult) -> list[str]:
    missing = []
    has_company = intent.company is not None and (
        (isinstance(intent.company, dict) and intent.company.get("value")) or
        (isinstance(intent.company, list) and len(intent.company) > 0)
    )
    if intent.query_type == QueryType.SINGLE_VALUE:
        if not has_company:
            missing.append("company")
        if not intent.metric:
            missing.append("metric")
        if not intent.time_range:
            missing.append("time_range")
    elif intent.query_type == QueryType.TREND:
        if not has_company:
            missing.append("company")
        if not intent.metric:
            missing.append("metric")
        if not intent.time_range:
            missing.append("time_range_for_trend")
    # 其他类型同理
    return missing
```

### 9.2 澄清问题生成流程

澄清问题生成包括三个步骤：

```
步骤1：模板选择
• 从澄清模板库选择对应槽位的模板
• 模板配置（chat.yaml:94-99）：
  company: "请问您想查询哪家公司的财务数据？"
  time_range: "请问您想查询哪个报告期的数据？（如：2025年第三季度、2024年度）"
  metric: "请问您想查询哪个财务指标？（如：利润总额、营业收入、净利润、总资产等）"
  time_range_for_trend: "请问您想查看哪个时间范围的趋势？（如：近三年、2023-2025年）"

步骤2：上下文丰富
• 根据已知槽位信息丰富澄清内容
• 若已知公司，补充"关于{公司名}"
• 若已知指标，补充"的{指标名}"
• 示例：
  原模板："请问您想查询哪个报告期的数据？"
  丰富后："关于金花股份的利润总额，请问您想查询哪个报告期的数据？"

步骤3：多槽位组合
• 若缺失多个槽位，组合多个澄清模板
• 用空格分隔，生成自然的连续问题
• 示例：
  缺失company和time_range
  澄清："请问您想查询哪家公司的财务数据？请问您想查询哪个报告期的数据？"
```

📍 **代码位置**：chat.py:977-1020

**丰富机制设计目的**：
- 提升用户体验：用户已知部分信息，无需重复提及
- 引导用户补全：提示已知信息，引导用户补充缺失信息
- 防止用户遗忘：若用户第二轮只补充部分信息，仍会保留已丰富的内容
