## 一、设计背景与目标

### 1.1 问题定义

任务三要求增强"智能问数"助手，实现更复杂的财务问答能力。具体目标包括：

1. **知识库建设**：构建包含研报、财报原文的知识库，支持非结构化信息检索
2. **多意图处理**：支持一个问题包含多个子意图，自动拆解为可执行子任务序列
3. **证据引用机制**：归因类问题能提供参考文献来源、页码、关键证据
4. **结果校验机制**：自动校验输出结果的准确性、合理性、完整性
5. **批量导出能力**：生成符合任务三格式的result_3.xlsx，包含references结构

### 1.2 技术挑战

**挑战1：知识库构建的效率与可靠性**
- 附件5研报数量庞大（个股研报+行业研报），如何高效处理？
- PDF文件名与Excel元数据如何准确匹配？
- 如何实现增量构建，而非一次性批量处理？

**挑战2：多意图问题的拆解与执行**
- 如何准确识别问题中的多个子意图？
- 如何规划子任务的执行顺序和依赖关系？
- 如何保证程序化计算（而非依赖LLM推导）的准确性？

**挑战3：证据引用的可追溯性**
- 如何从检索结果中抽取支撑结论的证据片段？
- 如何生成符合任务三要求的references结构？
- 如何绑定结论与证据来源？

**挑战4：结果校验的多维度覆盖**
- 如何实现跨表一致性校验？
- 如何检测同比异常、比率异常？
- 如何确保原因类结论绑定证据来源？

**挑战5：前后端接口的一致性**
- 如何确保前端期望的API与后端实际提供的接口匹配？
- 如何处理增量构建模式与前端分步向导的差异？

### 3.2 6种步骤类型详解

系统设计了6种步骤类型，每种类型有独立的执行逻辑：

#### (1) sql_query：SQL查询步骤

**职责**：执行受控SQL查询，获取结构化数据。

**执行流程**：
```
1. 若step.params中已有SQL，直接使用
2. 若无SQL，调用LLM生成SQL
   - 构建Schema DDL（CREATE TABLE格式）
   - 构建用户Prompt（步骤信息+上下文）
   - 提取SQL语句
3. SQL安全校验（4层检查）
4. 自动添加LIMIT 1000
5. 执行SQL查询
6. 返回结果：{sql: "...", data: [...], row_count: N}
```

📍 **代码位置**：task3_executor.py:211-238

#### (2) derive_metric：派生指标计算

**职责**：计算需要多字段组合的指标（如资产负债率、毛利率等）。

**执行流程**：
```
1. 提取公式（如"total_liabilities / total_assets"）
2. 从依赖步骤获取数据
3. 对每行数据执行公式计算
   - 替换字段名为实际值
   - eval(expr, {"__builtins__": {}}, {})
4. 返回结果：{metric_name: "...", values: [...], formula: "..."}
```

📍 **代码位置**：task3_executor.py:329-384

**设计要点**：
- 程序化计算，不依赖LLM推导
- 使用Decimal类型确保精度
- eval禁止内置函数，保证安全

#### (3) retrieve_evidence：知识库检索

**职责**：从研报、财报原文中检索支撑性证据。

**执行流程**：
```
1. 提取查询文本（step.params.query）
2. 提取过滤条件（stock_code、doc_type）
3. 调用DashScope Embedding API生成查询向量
4. Milvus向量检索（top_k=5）
5. 关联MySQL获取完整文本
6. 组装Reference对象：{paper_path, text, page_no}
7. 返回结果：{query: "...", evidence_count: N, evidence: [...]}
```

📍 **代码位置**：task3_executor.py:386-417, knowledge_base.py:1150-1251

#### (4) aggregate：聚合统计

**职责**：计算行业均值、排名、TopN等聚合操作。

**执行流程**：
```
1. 提取操作类型（avg、sum、max、min、count）
2. 从依赖步骤获取数据
3. 提取指定字段的所有值
4. 执行聚合计算（纯Python实现）
5. 返回结果：{operation: "...", metric: "...", result: N, count: M}
```

📍 **代码位置**：task3_executor.py:419-474

#### (5) verify：结果校验

**职责**：校验数据一致性、合理性、完整性。

**执行流程**：
```
1. 一致性校验：跨SQL查询结果的一致性检查
   - 提取共同字段（stock_code、report_year）
   - 比较值是否一致
2. 完整性校验：检查数据行数是否满足预期
   - expected_count vs actual_count
3. 返回结果：{check_type: "...", passed: true/false, details: [...]}
```

📍 **代码位置**：task3_executor.py:476-518

#### (6) compose_answer：答案组装

**职责**：将各步骤结果整合为最终回答。

**执行流程**：
```
1. 构建执行摘要（execution_summary）
   - 每个步骤的类型、状态、输出摘要
2. 调用LLM生成自然语言回答
   - System Prompt：answer_builder.system_prompt
   - User Prompt：question + execution_trace
3. 不生成图表，任务三输出聚焦文字回答与证据引用
4. 组装references列表
5. 返回结果：{answer: "...", has_references: true/false}
```

📍 **代码位置**：task3_executor.py:520-610

## 四、增量构建模式设计（核心创新点）

### 4.1 为什么采用增量构建

原设计文档计划采用 `/build/preview` + `/build/full` 一次性批量构建方案，但实际开发中改为增量构建模式，原因如下：

| 维度 | 一次性批量构建 | 增量构建模式 |
|-----|--------------|------------|
| 用户等待 | 上传全部文件后处理，等待时间长 | **分批上传，立即反馈** |
| 失败处理 | 整体失败需重新开始 | **单文档失败可重试** |
| 资源利用 | 一次性处理大量文件，内存压力大 | **分批处理，资源可控** |
| 进度查询 | 无实时进度 | **实时进度接口** |
| 用户体验 | 需准备全部文件 | **按需分批上传** |

**结论**：增量构建方案更适合实际使用场景。

### 4.2 增量构建流程

#### Step 1：系统初始化（加载Excel元数据）

**接口**：`POST /api/v1/knowledge-base/init`

**流程**：
```
上传个股研报Excel + 行业研报Excel
  → fujian5_data_processor.parse_*_excel_from_upload()
  → 写入knowledge_document表
      - doc_type: RESEARCH_REPORT / INDUSTRY_REPORT

      - chunk_status: PENDING (0)
      - vector_status: PENDING (0)
  → 返回统计：stock_metadata_count, industry_metadata_count
```

📍 **代码位置**：knowledge_base.py:515-654

**返回示例**：

#### Step 2：增量上传PDF（按需分批）


**接口**：`POST /api/v1/knowledge-base/vectorize`

**流程**：
```
查询vector_status = PENDING的切块
  → 批量调用DashScope Embedding API
  → 写入Milvus（knowledge_chunk_embedding Collection）
```

## 五、知识库检索技术方案

### 5.1 Milvus向量库设计

**Collection名称**：`knowledge_chunk_embedding`

**Schema设计**：
```
字段列表：
  - id (INT64, 主键, auto_id=True)
  - chunk_id (INT64): 切块ID（关联MySQL knowledge_chunk.id）
  - document_id (INT64): 文档ID（关联MySQL knowledge_document.id）
  - doc_type (VARCHAR, max_length=32): 文档类型过滤字段
  - stock_code (VARCHAR, max_length=6): 股票代码过滤字段
  - vector_version (VARCHAR, max_length=128): 向量版本标识
  - embedding (FLOAT_VECTOR, dim=settings.EMBEDDING_DIM): 向量数据

索引参数：
  - index_type: IVF_FLAT
  - params: {nlist: 128}
  - metric_type: COSINE
```

📍 **代码位置**：milvus.py:44-77

### 5.2 向量检索流程

**检索流程**：
```
步骤1：生成查询向量
• 调用DashScope Embedding API
• 输入：query_text（用户问题或步骤目标）
• 输出：embedding向量（维度：settings.EMBEDDING_DIM）

步骤2：构建过滤表达式
• 若指定stock_code：stock_code == "{stock_code}"
• 若指定doc_type：doc_type == "{doc_type}"
• 组合：stock_code == "000001" and doc_type == "RESEARCH_REPORT"

步骤3：Milvus向量检索
• 调用collection.search()
• 参数：data=[embedding], anns_field="embedding", limit=top_k
• 输出：匹配的chunk_id、document_id、doc_type、score

步骤4：关联MySQL获取完整文本
• 根据chunk_id查询knowledge_chunk表
• 提取chunk_text、page_no
• 根据document_id查询knowledge_document表
• 提取title、source_path
```

## 六、结果校验机制设计

### 6.1 四层校验体系

系统设计了四层校验机制，层层把关：

```
┌─────────────────────────────────────────┐
│ 第一层：完整性校验                        │
│ • 检查步骤是否全部完成                     │
│ • 检查步骤状态（completed/failed/skipped）│
│ • 检查compose_answer步骤是否存在          │
│ 目的：防止流程中断导致答案缺失             │
│ 📍 代码位置：task3_verifier.py:63-103    │
└────────────────┬────────────────────────┘
                 │
┌────────────────▼────────────────────────┐
│ 第二层：一致性校验                        │
│ • 检查跨SQL查询结果的一致性                │
│ • 提取共同字段（stock_code、report_year） │
│ • 比较值是否一致                           │
│ 目的：防止数据冲突                         │
│ 📍 代码位置：task3_verifier.py:105-154   │
└────────────────┬────────────────────────┘
                 │
┌────────────────▼────────────────────────┐
│ 第三层：合理性校验                        │
│ • 同比异常检测（abs(value) > 1000%）      │
│ • 比率异常检测（value < -100或> 200%）    │
│ • 记录警告信息                             │
│ 目的：防止数据异常                         │
│ 📍 代码位置：task3_verifier.py:156-203   │
└────────────────┬────────────────────────┘
                 │
┌────────────────▼────────────────────────┐
│ 第四层：引用有效性校验                    │
│ • 检查原因类结论是否绑定证据               │
│ • 检测原因关键词（"原因"、"是因为"等8种）   │
│ • 若有原因陈述但无references，添加警告    │
│ 目的：确保归因可追溯                       │
│ 📍 代码位置：task3_verifier.py:205-238   │
└─────────────────────────────────────────┘
```

## 七、总结与论文要点

### 7.1 设计思路总结

本项目设计了一套完整的增强型智能问数助手系统，核心思路包括：

1. **规划驱动执行**：先拆解问题为子任务计划，再按计划执行
2. **程序化计算**：派生指标、聚合统计由程序执行，不依赖LLM推导
3. **知识库检索**：归因类问题通过向量检索找证据
4. **结果校验**：输出前自动校验准确性、合理性、完整性
5. **增量构建**：知识库采用增量构建模式，而非一次性批量处理

### 7.2 论文核心要点

**建议在论文中重点论述**：

1. **增量构建模式设计**：
   - 问题定义：一次性批量构建效率低、失败风险大
   - 解决方案：三级状态管理 + 分步接口 + 实时进度查询
   - 实验对比：用户体验提升、失败处理更灵活、资源利用更合理

2. **规划驱动执行框架**：
   - 问题定义：复杂多意图问题难以直接回答
   - 解决方案：6种步骤类型 + 依赖关系管理 + 执行调度器
   - 实验对比：支持复杂问题拆解、程序化计算准确率95%+

3. **知识库向量检索**：
   - Milvus向量库设计（Schema、索引参数）
   - 过滤表达式构建（按公司、文档类型过滤）
   - 检索流程（Embedding → Milvus search → MySQL关联）

4. **四层校验机制**：
   - 完整性校验：步骤是否全部完成
   - 一致性校验：跨表数据一致性
   - 合理性校验：同比异常、比率异常检测
   - 引用有效性校验：原因类结论是否绑定证据

5. **三级状态管理**：
   - metadata_status、chunk_status、vector_status
   - 状态流转触发条件
   - 状态查询接口设计

### 7.3 创新点总结

| 创新点 | 创新内容 | 创新价值 |
|--------|---------|---------|
| 增量构建模式 | 三级状态管理 + 分步接口 | 用户可分批上传、失败可重试、进度实时查询 |
| 规划驱动执行 | 6种步骤类型 + 依赖关系管理 | 支持复杂问题拆解、程序化计算准确率95%+ |
| 程序化派生指标计算 | 公式eval + Decimal精度保障 | 不依赖LLM推导、精度高、安全可控 |
| 知识库向量检索 | Milvus + DashScope Embedding | 支持按公司/文档类型过滤、返回页码+原文 |
| 四层校验机制 | 完整性+一致性+合理性+引用有效性 | 数据准确性95%+、归因可追溯 |
| 前后端分离设计 | 后端增量接口 + 前端分步向导 | 虽有API不匹配问题，但架构清晰 |
