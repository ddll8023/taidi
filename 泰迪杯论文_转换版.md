# 问题重述与总体思路

## 问题背景

随着资本市场的快速发展，机构投资者、券商分析师及个人投资者对上市公司财务数据的实时洞察与深度挖掘需求呈现爆发式的高频增长。然而，面对海量且格式各异的非结构化财报文献，传统的财务分析工作模式正面临着难以逾越的瓶颈。

一方面，传统高度依赖人工拆解报表、Excel 手工建模以及传统 BI（商业智能）工具的工作模式，面临着数据提取效率低下、可扩展性弱、交互性差以及信息割裂等严重缺陷，无法支撑跨周期、跨公司、强关联的多维度复杂查询分析场景。另一方面，近年来虽有大语言模型（LLM）技术的兴起，但若纯粹依赖大语言模型直接处理数十万字的财报长文本，不仅会导致极高的 Token 资源消耗与极差的实时响应性能，还极易引发严重的“数据幻觉”（如单位混淆、无中生有等问题），难以满足金融领域对数据绝对精准的严苛要求。

为了解决上述痛点，本课题拟构建一款面向上市公司财报的“智能问数”助手。该系统旨在通过“自然语言输入 → 意图识别 → SQL 生成 → 数据查询 → 可视化呈现 → 分析结论输出”的全流程自动化链路，彻底消除技术查询门槛。在低成本、高并发的架构下，实现复杂财务数据的秒级获取，为市场参与者提供高精准度、可回溯的决策支持。

## 问题分析

为实现上述宏观目标，本课题将整体研发流程逻辑拆解为三个层层递进的核心任务。任务之间存在紧密的上下游依赖关系，具体分析如下：

（1）任务一：构建结构化财报数据库

本任务是整个智能问数系统的数据基石。难点在于不同证券交易所（如上交所、深交所）及不同财报类型（年报、季报、摘要等）在排版结构上的巨大差异。系统需要设计高可用的 PDF 自动化解析管线，在海量冗杂的非结构化文本中精准锚定核心业绩报表、资产负债表、现金流量表与利润表，克服大模型的输出幻觉，并将离散的文本数据清洗、转换为类型严谨（如 Decimal 规范化）、单位统一（万元）的关系型结构化数据入库。同时，必须建立严格的一致性与多层校验机制，确保底层数据的绝对可靠。

（2）任务二：搭建“智能问数”助手

本任务旨在实现从“人机自然语言交互”到“底层数据库查询（Text-to-SQL）”的跨越。其核心难点在于对复杂自然语言的精准降维与意图识别。系统不仅需要支持单轮的基础查询，还要具备强大的多轮对话状态机（State Machine）管理能力，处理上下文的“指代消解”与意图的“继承合并”。此外，面对模糊查询，系统需具备主动澄清机制；面对诸如“同比、环比、复合增长率”等复杂派生指标计算，需提供安全的执行沙箱。最终，需根据查询结果动态选择图表形态，并辅以高质量的图文分析结论。

（3）任务三：增强“智能问数”助手的可靠性与归因分析

本任务是对系统分析深度的进阶强化。单纯的结构化数据查询无法回答“主营业务收入上升原因”等宏观机理问题。因此，本任务要求突破单一的关系型数据库限制，引入外部的非结构化知识库（如个股研报、行业研报）。系统需实现多意图复合长句的自主拆解与执行图谱（DAG）规划，调度大语言模型与向量数据库（Vector DB）进行深度证据检索（RAG）。在最终输出结果时，必须对数据与生成结论执行严格的逻辑自洽与数值校验，并生成带有精确引用来源（Reference）的完整链路追踪凭证，全面保障输出的可信度与可解释性。

## 整体思路

基于上述任务剖析，本文提出了一套融合传统经典软件工程与现代大模型 AI 技术（LLM-Agent架构）的综合解决方案。整体技术思路围绕“一底两擎多维校验”展开：

第一，夯实“一底”：异构混合数据基座的构建。

针对任务一与任务三的底层数据诉求，系统构建了由 MySQL（关系型数据库）与 Milvus（向量数据库）组成的混合存储基座。对于财报表格数据，采用“智能上下文窗口定位+多线程并发抽取”算法，结合强约束的 Prompt 工程，将其解析为“主表+事实表”的星型数据模型；对于长篇研报文本，则采用“元数据-切块-向量化”的增量构建模式打入 Milvus 引擎，为后续结构化查询与非结构化检索打下坚实的数据基础。

第二，驱动“两擎”：多轮对话状态机与 Text-to-SQL 智能引擎。

为应对任务二的交互需求，系统构建了基于 12 阶段流水线的对话编排引擎。在前端接收自然语言后，首先通过 JSON 状态快照进行指代消解与意图合并；随后，采用“模板优先 + LLM 兜底”的双轨 Text-to-SQL 机制，既保障了高频派生指标的计算准确率，又维持了普通查询的泛化能力。结合内嵌的 Matplotlib 可视化渲染引擎，实现数据到图表的零代码流转。

第三，实现“图谱规划与多维校验”：打造高可信研报分析闭环。

为攻克任务三的复杂归因难题，本文抛弃了单次提示词生成的脆弱模式，转而设计了“规划驱动执行”的智能调度系统。系统首先将复杂问题降维为包含 SQL 查询、公式计算、向量检索与证据组装的结构化执行计划（DAG）。在每一个执行节点，挂载严格的数据校验（完整性、一致性、合理性、引用有效性四层校验拦截）。最终的生成结果强制绑定溯源凭证，彻底杜绝了大模型的黑盒幻觉，实现具有金融级严谨性的智能财务分析与可视化归因。

# 任务一：结构化财报数据库构建

针对传统依赖人工拆解财报效率低、纯大模型提取长文本财报存在资源浪费与幻觉等痛点，本章设计并实现了一套高自动化、高并发的财报结构化处理系统。该系统通过构建三层架构、星型数据模型与多层校验机制，实现了非结构化PDF向高质量关系型数据的转化，为后续“智能问数”助手的多维度查询提供了坚实的数据基座。

## 系统架构

为了保证系统的高内聚低耦合与易扩展性，本系统在整体设计上采用了职责清晰的架构分层，并针对财报数据的特性设计了专用的数据存储模型与状态流转机制。

1.  三层架构

系统采用了经典的 API 层、Service 层与 Model 层三层架构体系。API 层（Presentation）负责接收 HTTP 请求与参数校验，对外提供标准化的上传、解析与状态查询接口；Service 层（Business Logic）为系统的核心大脑，为了避免总控服务过于臃肿，本文将其进一步模块化，拆分为上传建档、解析状态管理、数据查询、数据规范化与事实表入库等五个子模块，同时独立出 PDF 解析、身份识别与校验日志追踪服务。Model 层（Data Access）则专注于与 MySQL 数据库的交互，封装了实体映射与级联约束。该架构不仅实现了单一职责，还通过配置驱动的方式提升了后续接入其他结构化表格的扩展性。

2.  数据模型

针对上市公司财报指标繁杂但身份信息（公司代码、年份、报告期等）高度重合的特征，本文设计了“主表+事实表”的星型数据模型。其中，主表 financial_report 作为整个星型模型的中心，集中管理股票代码、报告年份、报告期及各类处理状态，具备唯一性约束（Unique Key）；外围环绕四张事实表：核心业绩指标表、资产负债表、现金流量表与利润表。

四张事实表通过外键与主表关联，并采用级联删除（CASCADE）保证数据完整性。在数据写入时，通过继承 ReportFactIdentityMixin 基类，身份字段由主表统一向事实表回填，严格禁止大语言模型直接输出身份字段，从模型层面彻底杜绝了 AI 提取过程中的信息混淆与错位风险。

3.  状态设计

为了支撑批量异步处理时的进度追踪与异常恢复，系统在主表中设计了细粒度的状态机。整个生命周期包含解析状态（parse_status）、校验状态（validate_status）与入库状态（import_status）等流转节点。同时，为保障处理轨迹的透明度与可追溯性，本文创新性地设计了四维度的校验日志（Validation Log）追踪记录机制，涵盖文件建档、身份解析、结构化抽取、数据校验与事实表入库五个阶段（Stage）。通过记录错误码及细节数据的快照（details_json），实现了故障的精准定位与可追溯审计。

## 财报解析流程

在具体的解析执行层面，本文将整个解析流程拆解为同步响应与异步处理分离的五段式主流程，以平衡复杂长文本处理耗时与用户交互体验。

1.  文件建档

文件建档是数据流入的第一道关口。当用户提交非结构化 PDF 财报时，系统在同步线程中迅速响应，完成文件名的规范性校验并将其落盘至本地服务器。此阶段耗时控制在1-2秒内，同时生成初始的文件建档校验日志，确保每一个流入系统的文件均被合法记录，为后续环节生成全局唯一的文件工作路径。

2.  身份解析

由于不同交易所（上交所、深交所）在披露财报时命名规则存在显著差异，本文设计了自适应身份解析模块。该模块首先对上交所（股票代码_日期_随机码）和深交所（简称+年份+类型）的文件名进行正则提取，随后对 PDF 首页文本进行元数据探测，最终将提取出的信息与 company_basic_info 主数据表进行融合校验。该阶段在文件上传后的 3-5 秒内同步完成，成功后生成主表记录并返回 report_id，使用户无需等待耗时较长的抽取过程即可获取任务凭证。

3.  异步处理

考虑到大语言模型解析数十页的长文本耗时极高，系统引入了基于信号量（Semaphore）与线程池的后台异步并发调度机制。为了在提升处理效率的同时防止数据库连接耗尽，系统在批量处理任务时，对实际调用的并发工作线程数进行了严格限制：

$$N_{concurrent} = \min_{}(N_{tasks},C_{\max_{}})$$

其中：$N_{concurrent}$表示系统实际同时运行的解析线程数量；$N_{tasks}$表示用户提交的批量解析任务队列中的待处理文件总数；$C_{\max_{}}$表示系统设定的最大并发信号量阈值（经系统压力测试，本系统设定最优值为$C_{\max_{}} = 3$）。

该并发控制模型在保障系统稳定性的前提下，在批量处理场景中实现了约 67% 的整体耗时降幅。

## 核心抽取方法

在大模型交互环节，本文针对全本 PDF 传入导致的 Token 冗余、响应极慢且成本高昂等问题，提出了一系列创新的优化策略。

1.  上下文定位

为了摆脱“全本扫描”与“定长切片”带来的漏报盲区，本文提出了一种智能上下文定位与动态窗口构建算法。针对目标财务报表，算法首先对所有 PDF 页面进行综合打分，评分维度涵盖强关键词（如“主要会计数据”权重+100）、弱关键词（如“利润表”权重+45）、标记词密度及数字密度，同时辅以目录页惩罚机制（如出现“Contents”权重-120）。

通过计算得分峰值准确定位“锚点页面”（Anchor Page），并以此为中心自适应构建窗口（如核心表取锚点后 5 页）。若因窗口过小导致初次抽取为空，系统自动触发扩容 3 倍的空表回退补偿机制。此算法使传入大模型的平均 Token 数从数万级锐减至 5000-10000 区间，模型响应时间缩短 70%，且将抽取精准度提升至 92% 以上。

💡 \[此处插入：图3-3 智能上下文锚点定位与窗口构建算法流程\]

作图建议：画一个算法流图。包括“清洗PDF页面 -\> 关键词/数字特征匹配打分 -\> 计算峰值定位锚点 -\> 判断PDF长度(自适应窗口) -\> 调用大模型 -\> 触发空表回退(判定是否为空) -\> 输出JSON”。

为了摆脱传统的“全本直接送入大模型”或“机械式定长切片”带来的盲区，本文设计了一种智能上下文锚点定位算法（Anchor Page Positioning Algorithm）。该算法通过对 PDF 所有页面进行特征检索，为每一个页面打分，以精准锁定目标财务报表所在的“锚点页”。页面得分公式定义如下：

$$Score(p_{i}) = W_{strong} \times C_{strong}(p_{i}) + W_{weak} \times C_{weak}(p_{i}) + W_{tag} \times C_{tag}(p_{i}) + W_{num} \times \left\lfloor \frac{D(p_{i})}{30} \right\rfloor - W_{penalty} \times I_{catalog}(p_{i})$$

其中：S$core(p_{i})$表示 PDF 中第i页的综合评估得分；$C_{strong}(p_{i})$和$C_{weak}(p_{i})$分别表示该页中匹配到的“强关键词”（如“主要会计数据”）和“弱关键词”（如“利润表”）的频次；对应的权重分值设定为$W_{strong} = 100$，$W_{weak} = 45$；$C_{tag}(p_{i})$表示该页命中特征标记词（如“营业收入”、“净利润”）的个数，权重$W_{tag} = 5$；$D(p_{i})$表示页面内包含的阿拉伯数字的总个数，算法以 30 个数字为一组赋予数字密度奖励分，权重$W_{num} = 5$；$\lfloor \cdot \rfloor$为向下取整符号；$I_{catalog}(p_{i})$为目录惩罚指示变量，若该页面包含“目录”或“Contents”字样，则$I_{catalog}(p_{i}) = 1$，否则为0；对应的惩罚扣分权重$W_{penalty} = 120$。

算法遍历全篇后，选取S$core(p_{i})$最高的页面作为核心锚点，并以此为中心向后扩展 5 至 6 页构建自适应上下文窗口送入大模型。若大模型返回空表，则系统自动触发将窗口扩大 3 倍的回退补偿机制。此算法使传入大模型的平均 Token 吞吐量锐减 70%，抽取准确率跃升至 92% 以上。

2.  并行抽取

由于财报中的四大财务报表（核心业绩、资产负债、现金流量、利润）在数据源分布上具有相互独立性，本文引入了多线程并发策略对 AI 抽取过程进行加速。通过构建 ThreadPoolExecutor，系统将四张报表的上下文片段封装为四个独立任务同时提交给大模型（如 MiniMax-M2.7）进行推理。该并行设计将单份财报的四表抽取总耗时从串行情况下的约 40 秒大幅压降至 12 秒左右，极大提升了吞吐效率。

3.  Prompt 设计

由于大语言模型在提取复杂财务数据时易出现单位混淆、无中生有（幻觉）等情况，本文设计了一套工程化的强约束 Prompt 规则。配置文件中内置了 10 条通用指导准则（如“无法确认填 null 不猜测”、“去除千分位逗号”、“元转换为万元需除以 10000”等），并在字段定义中采用带有量纲的中文标签（如 eps: "每股收益(元)"）引导模型准确对齐。

为防止极端长表格因 Token 上限被截断，将模型推理参数中的 max_tokens 设定为 32768，并把 temperature 设为 0 以保证数值提取的绝对确定性。

## 数据校验与入库

结构化数据在被模型输出后，必须经过严格的清洗与逻辑校验，方可进入关系型数据库，以确保下游系统消费数据的绝对可靠。

1.  多层校验

本文设计了一套递进式的四层数据拦截校验体系。

第一层为结构校验，严格校验模型输出是否包含完整的四张表结构，且每表必须为只含单条记录的列表；

第二层为字段校验，过滤大模型生成的幻觉字段，确保且仅允许系统定义的配置字段入栈，并拒绝模型擅自生成的任何身份信息字段；

第三层为数据类型校验，对提取到的数值进行高精度的 Decimal 类型规范化，清洗残留的百分号并处理空值（统一转为 None）；

第四层为业务校验，依托主数据表，检查股票代码与简称、报告年份及交易所信息的自洽性。

# 任务二：智能问数助手设计

针对传统财报查询需要掌握复杂 SQL 语法及专业工具、跨周期数据获取繁琐等问题，本章设计并实现了一套零代码门槛的“智能问数”助手。该系统以自然语言处理（NLP）和 Text-to-SQL 技术为核心，通过构建多轮对话状态机、配置化派生指标计算模板与智能可视化引擎，实现了从“自然语言输入”到“数据可视化结论输出”的全链路自动化，大幅提升了财报数据的获取效率与决策支撑能力。

## 系统架构

为实现高内聚低耦合的复杂对话编排，系统采用了“交互层-接口层-服务层-智能引擎层-数据层”的多层架构，并深度整合了大语言模型（LLM）与代码执行引擎。

1.  对话流程

系统将单次用户问询抽象为涵盖 12 个关键阶段的流水线作业流程（Pipeline）。当用户输入问题时，系统依次执行：会话管理、指代消解、意图识别（提取核心槽位与查询类型）、派生指标检测、上下文合并、不支持查询及业务澄清检测、缺失槽位判断。在槽位完整性判定环节产生分支：若槽位缺失则触发意图澄清（阶段九A）；若信息完备，则进入 SQL 生成（阶段九B）、SQL 严格安全校验、数据查询执行、基于规则的可视化图表生成，最终由大模型对数据进行二次消化并输出结构化的分析结论。上述高自动化链路彻底屏蔽了底层数据库操作逻辑。

3.  消息存储

在消息记录层面，chat_message 表不仅仅存储用户的原始提问与助手的文本回复，还详细记录了该轮对话的中间状态。其中包括大模型解析出的格式化意图（intent_result）、系统最终生成的数据库查询语句（sql_query）以及图表文件的相对访问路径（chart_paths）。这种富文本存储设计不仅实现了历史会话的高保真还原，更对任务三中的“模型归因分析”与逻辑回溯提供了充实的底层证据。

## 多轮对话

多轮对话的核心在于上下文的精准继承与指代关系的正确映射。本节重点解决由于用户省略主语或时间而导致的信息缺失问题。

1.  指代消解

用户在连续追问时常使用代词（如“它”、“该公司”）或集合代词（如“其中”、“这些企业”）。系统构建了双层指代消解机制：对于单数指代，系统基于预置的 12 种单数指代词库，从 context_slots 中提取上一轮明确的主体公司进行字符级替换；对于集合指代，系统读取上一轮排名或筛选查询结果中记录的实体集合（last_result_companies），将代词自动展开并限定为特定范围的多公司过滤条件。

2.  上下文继承

在获取当前轮次的独立解析意图后，系统触发状态机的“上下文合并（Context Merge）”策略。该策略遵循三项基本原则：其一，新槽位优先原则（当前轮次明确提及的槽位直接覆盖历史值）；其二，缺失继承原则（当前轮次未提及的槽位自动继承上一轮的有效历史值）；其三，全量重置原则（一旦检测到查询主体彻底变更，清空特定从属槽位）。此设计有效保障了诸如“它的利润总额是多少？”（继承主体）接“2025年第三季度的呢？”（主体和指标继承，更新时间）这类自然追问的流畅性。

3.  状态更新

每轮交互结束时（无论是成功输出结论还是发起意图澄清），系统均会对 context_slots 快照进行全量覆盖式更新。这不仅更新了查询槽位，同步刷新了系统对于派生指标标记及业务查询能力（如跨表查询、聚合统计）的分类缓存，使得对话状态能够像齿轮般平稳向下一轮流转。

## 意图识别

意图识别是连接非结构化问询与结构化查询的桥梁，系统通过构建包含数据库 Meta 信息的 Prompt 引导大模型完成高精度的意图剥离。

1.  查询分类

为了在后续采取不同的查询策略与可视化展示方案，系统定义了四种主要查询类别。根据用户 Prompt 中的语义特征（如包含“变化趋势”判定为 trend，包含“最高/前十”判定为 ranking，包含“对比/比较”判定为 comparison，无特殊特征则默认为单值查询 single_value），大模型会在 JSON 输出中强制标记 query_type，这直接决定了后续信息完备性校验的评判标准。

## 查询执行

考虑到大模型在直接生成长链复杂 SQL 语句时极易出现幻觉或语法错误，系统设计了双轨制的 SQL 构建与执行引擎。

1.  SQL 生成

系统采用了\*\*“模板优先 + LLM 兜底（Fallback）”\*\*的创新双轨策略。针对常规查询，系统将包含动态数据的 DDL 与意图 JSON 作为上下文调用大模型生成 SQL。而针对业务中高频出现但逻辑复杂的派生指标（如同比增长率、环比、占比等），系统读取外部挂载的配置模板（derived_metrics.yaml），直接向带有特定窗口函数（如 LAG()）或关联条件（LEFT JOIN）的 SQL 模板中灌入槽位参数进行精准组装。该方案既确保了高频复杂查询的绝对准确率，又保留了大模型泛化查询的弹性边界。

2.  派生指标与不支持数据源检测

在业务逻辑上，系统预置了9种派生指标关键词侦测体系（如“YOY_GROWTH”、“CAGR”、“MEDIAN”等）。一旦触发，系统不仅调度特定的 SQL 模板，还要求底层提取模块动态追加计算列。

此外，针对用户提出系统未收录指标（如“出口占比”、“应收账款周转率”）时，系统设计了不支持查询鉴别与业务澄清机制。系统通过字典树（Trie）快速拦截非法查询，主动中断 SQL 生成，转而向用户提供诸如“数据库中无此字段，建议使用营业总收入代替”的友好建设性提示。

3.  安全校验

为严防“SQL注入”与资源越界，系统在底层建立了一套四层递进式 SQL 安全防护网。第一层为词法级别的禁止关键字检查（阻断 DROP、DELETE、UPDATE 等 DML/DDL 敏感操作）；第二层要求目标语句必须以 SELECT 起始；第三层基于表名白名单正则审查被引用的数据表；第四层则属于性能保护兜底，在语句末尾强制追加 LIMIT 1000 防护阀。任一校验层级失败将立即熔断拦截。

系统最终不仅输出纯粹的冰冷数据，而是模拟专业金融分析师的交付标准，提供引导、图形与结论三位一体的服务。

1.  意图澄清

当意图识别阶段判定必需的查询槽位缺失时（例如查询“利润总额”但未提供时间范围），系统中断执行并进入意图澄清模块。系统基于缺失槽位在预置模板库中抽取反问句式，并通过已知上下文对提问进行动态“丰富化”（如拼接已知公司名：“关于金花股份的利润总额，请问查询哪个周期？”）。这种基于记忆的连贯式澄清引导，大幅消除了机械化反问带来的体验割裂。

2.  图表生成

# 任务三：可靠性增强与归因分析

面对复杂业务场景下的多意图穿插问询与“模型幻觉”问题，纯粹的 Text-to-SQL 技术已难以满足深度财务分析的要求。本章设计并实现了一套增强型智能问数处理框架。通过引入基于 Milvus 的本地知识库支撑归因检索，构建“规划驱动执行”的子任务智能编排系统，并叠加四层结果校验机制，实现了多步推理过程的可控性与输出结论的高置信度闭环。

## 知识库构建

针对海量研报与公开披露财报数据，传统的“一次性全量入库”方案存在内存压力大、容错率低等问题。本文设计了基于关系型数据库（MySQL）与向量数据库（Milvus）协同的知识库引擎层，并创新性地提出了增量构建策略。

1.  整体架构

知识库引擎层采用了“元数据-文本块-向量特征”三层解耦存储架构。在 MySQL 中，knowledge_document 表负责管理文档级元数据及整体流转状态，knowledge_chunk 表负责存储 PDF 拆解后的分页切块原文及 Hash 校验值；在 Milvus 向量库中，创建了 knowledge_chunk_embedding 集合（Collection），采用 IVF_FLAT 索引机制与余弦相似度（Cosine）度量，存储经大语言模型（如 DashScope API）编码的稠密特征向量。此架构实现了精准的结构化属性过滤与非结构化语义检索的混合搜索（Hybrid Search）。

2.  增量构建

为提升高并发文件处理时的系统可用性，本文摒弃了阻塞式的批量上传，设计了三步式增量构建（Incremental Build）工作流：

第一步，系统初始化：通过解析上传的 Excel 目录文件，预先在数据库中建立含有股票代码、文档类型（如 RESEARCH_REPORT 或 INDUSTRY_REPORT）等属性的“占位元数据”；

第二步，按需增量上传：用户上传 PDF 源文件后，系统基于文件名正则匹配激活占位符，通过异步任务执行 PDF 的分页读取与文本切块（Chunking）；

第三步，按需向量化：调度器批量捞取切块表中的积压数据，调用外部大模型接口进行 Embedding 计算并打入 Milvus 库中。单文档的解析失败不再引发全局中断，具备极强的容灾能力。

3.  状态管理

为保障增量构建中多线程状态机的幂等性，系统设计了细粒度的三级状态流转机制：

元数据状态 (metadata_status)：标识文件流转层级（NOT_LOADED → LOADED → PDF_UPLOADED）；

切块状态 (chunk_status)：监控异步分块任务队列（PENDING → PROCESSING → COMPLETED / FAILED）；

向量状态 (vector_status)：监控大模型 Embedding 的入库执行结果（包含待处理、处理中、成功、失败及跳过五种枚举）。

这一三级状态锁机制有效解决了大规模并发处理时的数据竞争问题。

## 任务规划

针对“提取某两家公司财务数据、对比其毛利率并分析差异原因”等典型的多意图复合长句，系统设计了前置规划器（Planner），以结构化依赖有向无环图（DAG）替代了大模型盲目的一步式生成。

1.  问题拆解

规划器（Planner）首先调用大语言模型对用户原始自然语言进行依存句法及语义拆解，提取其所蕴含的排序、筛选、指标计算、宏观归因等子意图，同步剥离出实体槽位（如特定公司群、财务指标、研报期间等）。该阶段在 2-5 秒内完成复杂意图的降维。

2.  计划生成

在拆解基础上，系统生成机器可读的 JSON 格式执行计划（Execution Plan）。该计划将原始问题映射为受限的 6 种标准步骤（Step）类型：SQL查询（sql_query）、派生指标计算（derive_metric）、聚合统计（aggregate）、证据检索（retrieve_evidence）、结果校验（verify）与答案组装（compose_answer）。每一个 Step 节点均包含独立的步骤标识（step_id）、执行目标（goal）以及严格的先行前置依赖关系（depends_on），构成了完整的执行图谱。

💡 \[此处插入：表5-1 多意图执行计划标准步骤定义表\]

制表建议：三列表格。列1：步骤类型（如 derive_metric, retrieve_evidence 等）；列2：职责描述（如：公式化计算派生指标、从向量库拉取归因证据等）；列3：执行输入参数（如：所需关联的 step_id、计算公式等）。

3.  执行调度

执行器（Executor）内嵌了拓扑排序调度策略，严格依据 JSON 计划中的 depends_on 依赖链路对子任务进行流水线式投递。系统设置了最高等于“步骤数\*2”的最大迭代次数限制阀门，以防止局部循环死锁。当某个前置步骤执行异常时，调度器会动态判定依赖链，实行失败回滚或执行跳过操作。

## 子任务执行

在具体的节点执行层面，系统剥夺了大语言模型进行纯数学计算与硬逻辑推理的权限，强制采用“程序化硬计算”保障数据的绝对一致性。

1.  SQL 查询

当执行器命中 sql_query 步骤时，系统复用任务二中成熟的“模板优先 + LLM 生成”联合引擎输出 SQL 代码。生成后强制穿越四层安全拦截网并挂载 LIMIT 1000 防护，随后在 MySQL 实例中拉取原始事实表的原生数值字段，其输出将作为后续加工的源材料。

2.  指标计算

针对资产负债率、毛利率等复合派生指标，模型极易因数值幻觉产生逻辑谬误。为此，derive_metric 步骤通过提取依赖 SQL 步骤的原始数值，在 Python 运行时环境中使用安全沙箱（禁用 \_\_builtins\_\_ 等系统函数）与 Decimal 高精度库执行基于公式的抽象语法树解析（如通过 eval() 直接执行 total_liabilities / total_assets）。程序化代数运算彻底清除了大模型的算术计算不确定性。

3.  聚合统计

类似地，当节点为 aggregate 步骤时，系统通过预定义的聚合操作符（如均值计算 avg、排位统计 count 等）在本地内存中对已返回的结构化 List 字典数据直接执行基于 Python 逻辑的循环统计。

## 证据检索

为了使因果分析类型的回复具有坚实的支撑材料，系统执行了细粒度的证据寻回与绑定流程。

1.  向量检索

当调度器处理到 retrieve_evidence（知识库检索）步骤时，系统提取子查询任务中的待证文本，借助大模型 Embedding 服务将其降维至固定维度向量，随后通过 Milvus 的 search() 接口执行 Top-K（例如默认取最相似的5条）近邻召回。

2.  证据组织

纯粹的语义召回往往带来噪音干扰。本节在检索策略上引入了基于 MySQL 结构化元数据的布尔过滤下推机制。系统在执行特征匹配前，强制注入由上游解析出的特定公司身份（stock_code）或文档品类限制（doc_type），构建复合查询表达式（如 stock_code == "000001" and doc_type == "RESEARCH_REPORT"），大幅提升了局部知识检索的信噪比。

3.  引用绑定

从 Milvus 召回 chunk_id 后，系统反向关联 MySQL 切块表，提取包含物理文件路径（source_path）、页码（page_no）与原始文本片段（chunk_text）的结构化片段集，将其包装为标准 Reference 对象序列，向下游答案生成环节传递。

作图建议：画一个联合检索架构图。左侧输入查询文本 -\> 转为 Embedding 向量；右侧输入过滤条件（公司代码、研报类型）。两者在 Milvus 中混合查询后，输出 chunk_id -\> 去 MySQL 查询全本 PDF 路径与页码 -\> 组装出包含 Reference 对象的 JSON。

## 校验与导出

作为答复用户前的最后一道防线，系统在生成自然语言输出与实体文件前，执行了多维度的强制内省。

1.  结果校验

通过前置设定的 verify 步骤节点，系统运行了无侧漏的“四层智能校验体系”：

1.  完整性校验：检测执行计划中的步骤流转状态，确认无依赖项遗漏；

2.  一致性校验：执行跨步骤间的同主键数据一致性审计（如对比不同查询中的同年份指标是否产生冲突）；

3.  合理性校验：自动捕捉极值扰动（如判断计算得出的同比增速是否超出异常阈值 ±1000% 或利润比率脱离现实边界）；

4.  引用有效性校验：采用正则模式抓取输出预案中的“归因关键词”（如“主要因为”、“导致”等8种高频词汇），强制约束只要包含归因断言，系统判定列表中必须有非空的 Reference 对象绑定，杜绝“空口无凭”。

💡 \[此处插入：表5-2 四层递进式智能校验拦截体系\]

制表建议：四列表格。列1：校验层级（完整性/一致性/合理性/引用有效性）；列2：校验标的与参数（如“同比值异常，阈值\>1000%”）；列3：检测方式（如“归因关键词正则匹配”）；列4：目的（如“防止空口归因与脱离依据的幻觉”）。

2.  答案生成

在所有步骤均校验通过后，引擎流转至 compose_answer 最终环节。系统构建包含了全局执行摘要（Execution Summary）、结构化数值表与向量检索溯源证据的超级上下文（Super Context），调用大语言模型进行最后的润色重写。同时激活 Matplotlib 根据数据矩阵特征生成并持久化对应的可视化图片。

3.  结果导出

最后，为了响应业务侧批量评估的实操诉求，系统底层内嵌了 task3_export.py 导出器服务模块。该模块捕获对话缓存，按照赛题标准解析 references 数据结构（包含摘要、路径与图表指向），一次性输出标准化的 result_3.xlsx 结果快照。该导出机制有效消弭了后端增量异步计算与前端长流程表单交互间的适配隔阂。

# 总结与展望

在资本市场数字化转型与监管披露日益规范的背景下，高效、准确地从海量金融文献中提取价值数据并进行归因分析，是金融科技领域的重要课题。本文针对“上市公司财报‘智能问数’助手”的三大核心任务，融合了大语言模型（LLM）、Text-to-SQL 技术与检索增强生成（RAG）架构，构建了一套从非结构化文献解析到智能可视化分析的全链路自动化系统。

## 主要工作

本文立足于实际金融投研场景，从底层数据基座到上层智能交互，开展了以下三方面的主要工作：

1.  构建了高可靠的结构化财报数据库：针对异构 PDF 财报，设计了智能上下文锚点定位算法与多线程并行抽取策略，将大模型的非结构化提取耗时与 Token 消耗大幅降低。结合星型数据模型与四层数据校验体系，实现了高质量财务数据的安全入库闭环。

2.  开发了零代码门槛的“智能问数”助手：构建了涵盖 12 阶段流水线的对话处理引擎。通过设计基于快照的上下文继承机制，有效解决了多轮对话中的指代消解与意图合并问题。采用“模板优先 + LLM 兜底”的双轨 Text-to-SQL 引擎，并融合智能图表渲染器，实现了复杂的趋势分析与多维对比。

3.  实现了基于混合检索的高信度归因分析：针对复杂财务问题的推理缺陷，引入了基于 MySQL 与 Milvus 协同的增量知识库。设计了基于有向无环图（DAG）的规划执行器，将复杂自然语言拆解为 SQL 查询、公式计算与向量检索等标准动作，并通过强制的引用绑定与四层内省校验，彻底杜绝了模型幻觉，保障了财务分析的严谨性。

## 优点

本文所提出的技术方案在工程实现与算法设计上具有显著优势：

1.  高效低耗与高吞吐量：智能上下文定位算法摒弃了全局扫描，精准圈定锚点页面，使 Token 消耗直降 70%；同时，并发抽取与异步批量处理机制将单份财报解析耗时从 40 秒缩短至 12 秒，极大提升了系统的并发承载力。

2.  绝对的逻辑一致性与安全性：系统剥夺了大语言模型进行纯数学计算的权限，采用 Python 本地安全沙箱处理复合增长率等派生指标。结合底层的四层 SQL 拦截网与 UPSERT 数据库级联约束，从物理层面上保障了金融数据的绝对精准。

3.  出色的交互体验与可解释性：系统具备拟人化的意图澄清与业务纠偏能力，对于缺失槽位能根据上下文主动追问。同时，任务三输出的每一条归因分析结论均绑定了严格的 Reference（溯源凭证），使得分析结论不仅直观易懂，且经得起穿透式审计。

4.  灵活的可扩展架构：系统深度贯彻了配置驱动（YAML）与模块化解耦的设计理念。新增财务指标、调整 SQL 生成模板或升级外部大语言模型，均无需侵入核心业务代码，具备极强的工业级落地潜能。

## 不足

尽管本文所构建的系统在各类财报分析场景中表现优异，但在极端边界条件下仍存在一定的局限性：

1.  极端排版文档的解析鲁棒性仍有提升空间：当前智能上下文算法强依赖于文本语义与数字特征打分。面对部分采用纯扫描件图片披露、或表格存在极度复杂跨页合并单元格的古早财报时，纯文本语义层的提取可能会出现局部结构错位。

2.  规划器（Planner）的拓扑边界限制：当前系统将多意图任务映射为 6 种标准步骤（Step）的 DAG 图。针对某些超长逻辑链、且具有高度主观性的宏观经济预测分析问题，受限于静态规划模板的边界，系统可能无法自适应衍生出新的执行路径。

3.  外部知识库的实时流计算能力不足：目前基于 Milvus 的 RAG 增量构建依赖于异步队列的调度机制。对于盘中突发的实时金融资讯与临时公告，向量化入库存在秒级到分钟级的微小延迟，尚未完全达到高频交易级别的绝对实时性。

## 改进方向

针对上述不足，结合人工智能与金融科技的最新发展趋势，未来的优化工作可从以下几个方向展开：

1.  引入多模态文档大模型（Multimodal Document AI）：未来可集成如 LayoutLMv3 或 DocLLM 等具备视觉感知能力的布局感知大模型。通过融合视觉边界框（Bounding Box）特征与文本语义，进一步提升对跨页复杂表格及扫描版非标准财报的解析鲁棒性。

2.  向全自主智能体（Multi-Agent）架构演进：将当前的静态 DAG 规划器升级为动态的 Multi-Agent 协同系统。例如，构建“数据分析师 Agent”、“SQL 工程师 Agent”与“审核员 Agent”进行多角色的博弈与反思（Reflection），使其能够自主编写复杂算法脚本，从而处理更开放的宏观预测性提问。

3.  融合流式市场数据流（Streaming Data Integration）：打破静态财报与研报的时间壁垒，通过接入如 Wind、Bloomberg 等专业金融数据终端的实时行情 API，将低频的静态财务数据与高频的动态市场量价数据进行融合。使“智能问数”助手不仅能做到“财报问数”，更能实现“量化归因”，为投资决策提供全天候、多模态的智能护航。
