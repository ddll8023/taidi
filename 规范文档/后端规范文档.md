# 后端 AI 编码规范

## 0. 目标

本规范用于约束 AI 生成后端代码时的默认行为。
除非明确说明，AI 生成代码时必须优先遵守本规范。

本规范的目标是：

- 保证项目内只有**一套明确的后端实现方式**
- 禁止多种命名风格、多种异常处理方式、多种返回方式并存
- 在 AI 生成代码时，避免“看起来都能用，但实现彼此不一致”的情况

---

## 1. 全局输出规则

### 1.1 统一响应格式

所有接口必须返回统一 JSON 结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

规则：

- `code=0` 表示成功，非 `0` 表示失败
- `message` 为简短状态描述
- `data` 为业务数据
- 失败时 `data` 默认为 `null`
- 不允许返回不含 `code/message/data` 的 JSON

---

### 1.2 分页响应格式

列表接口必须使用以下分页结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "lists": [],
    "pagination": {
      "page": 1,
      "page_size": 10,
      "total": 100,
      "total_pages": 10
    }
  }
}
```

规则：

- 列表接口统一返回 `ApiResponse[PaginatedResponse[T]]`
- `lists` 和 `pagination` 均位于 `data` 内部
- `page` 从 `1` 开始
- `page_size` 为每页数量
- `total` 为总记录数
- `total_pages` 为总页数
- 列表接口不得返回裸数组
- API 层 `response_model` 声明为 `ApiResponse[PaginatedResponse]`，泛型 `T` 由内层列表项类型填充

---

### 1.3 响应构造唯一入口

API 层对外返回时，**只能**使用统一 helper：

- 成功：`success(data=..., message="success")`
- 失败：`error(code=..., message=..., data=None)`

标准实现如下：

```python
from typing import TypeVar
from app.schemas.common import ErrorCode

T = TypeVar("T")


def success(data: T | None = None, message: str = "success") -> dict:
    return {"code": ErrorCode.SUCCESS, "message": message, "data": data}


def error(
    code: int = ErrorCode.INTERNAL_ERROR,
    message: str = "服务器错误",
    data: T | None = None,
) -> dict:
    return {"code": code, "message": message, "data": data}
```

规则：

- API 层不得手写零散 `{"code": ..., "message": ..., "data": ...}`
- API 层不得混用 `JSONResponse`、裸 `dict`、`ApiResponse(...)` 多套返回方式
- **唯一方式**：统一调用 `success()` / `error()`
- 虽然 helper 返回的是 `dict`，但其结构必须符合 `ApiResponse` 定义
- **统一放置路径**：`app/schemas/response.py`
- 使用方式：`from app.schemas.response import success, error`

---

### 1.4 通用响应模型唯一标准

统一响应模型定义如下：

```python
from typing import Generic, TypeVar
from enum import IntEnum
from pydantic import BaseModel, ConfigDict, Field

T = TypeVar("T")


class ApiResponse(BaseModel, Generic[T]):
    code: int = 0
    message: str = "success"
    data: T | None = None

    model_config = ConfigDict(
        from_attributes=True,
        json_schema_extra={"example": {"code": 0, "message": "success", "data": {}}},
    )


class PaginationInfo(BaseModel):
    page: int = 1
    page_size: int = 10
    total: int = 0
    total_pages: int = 0


class PaginatedResponse(BaseModel, Generic[T]):
    lists: list[T] = Field(default_factory=list)
    pagination: PaginationInfo = PaginationInfo()

    model_config = ConfigDict(from_attributes=True)


class ErrorCode(IntEnum):
    SUCCESS = 0
    PARAM_ERROR = 1001
    DATA_NOT_FOUND = 1002
    NOT_LOGGED_IN = 2001
    TOKEN_EXPIRED = 2002
    PERMISSION_DENIED = 2003
    UNSUPPORTED_FILE_FORMAT = 3001
    FILE_TOO_LARGE = 3002
    AI_SERVICE_ERROR = 4001
    INTERNAL_ERROR = 5001
    PASSWORD_ERROR = 6001
```

规则：

- `ApiResponse`、`PaginationInfo`、`PaginatedResponse`、`ErrorCode` 为通用原语
- 不允许在多个文件重复定义同义版本
- 代码中必须使用 `ErrorCode.xxx`，禁止硬编码数字错误码
- 当前项目统一保留 `AI_SERVICE_ERROR = 4001` 这一命名，不再派生第二套 4001 命名

---

### 1.5 错误码规则

固定错误码：

| code | 含义           |
| ---- | -------------- |
| 0    | 成功           |
| 1001 | 参数错误       |
| 1002 | 数据不存在     |
| 2001 | 未登录         |
| 2002 | Token 过期     |
| 2003 | 权限不足       |
| 3001 | 文件格式不支持 |
| 3002 | 文件过大       |
| 4001 | 服务调用失败   |
| 5001 | 系统内部错误   |
| 6001 | 账号密码错误   |

错误码分段：

- `1xxx`：业务/参数类错误
- `2xxx`：认证/授权错误
- `3xxx`：文件相关错误
- `4xxx`：外部服务相关错误
- `5xxx`：系统级错误
- `6xxx`：登录认证类错误

规则：

- AI 不得随意新增错误码段
- 已有错误优先使用固定错误码
- 认证失败必须使用 `2001/2002/2003/6001`
- 代码中使用 `ErrorCode` 枚举，不写裸数字
- `4001` 在本项目中统一对应 `ErrorCode.AI_SERVICE_ERROR`
- 对于 LLM、Milvus、向量库、外部 API 等外部依赖失败，统一归并到 `4001`

## 2. 分层架构规则

### 2.1 分层结构

项目采用四层结构：

- `app/models/`：数据模型层
- `app/schemas/`：数据校验与序列化层
- `app/services/`：业务逻辑层
- `app/api/` 或 `app/routers/`：接口路由层

constants 层与 services 层对齐，用于集中管理 services 层的常量。只有含有常量的 service 才创建对应的 constants 模块，文件命名与 service 文件名一致。

#### Services 文件组织规则

一个 API 模块对应一个 services 模块。根据模块复杂度，services 文件组织遵循以下规则：

- **单文件模块**：模块逻辑简单、代码量较少时，直接在 `app/services/` 下放置单个 `.py` 文件
- **多文件模块**：模块逻辑复杂、单文件代码量较大时，在 `app/services/` 下新建以模块业务名命名的文件夹，将拆分后的 service 文件放入其中

单文件示例：

```text
app/services/
├── chat.py
├── knowledge_base.py
└── financial_report.py
```

多文件示例：

```text
app/services/
├── task2/
│   ├── __init__.py
│   ├── runner.py
│   ├── export.py
│   └── workspace.py
└── task3/
    ├── __init__.py
    ├── runner.py
    ├── planner.py
    └── executor.py
```

规则：

- 拆分为子目录时，必须包含 `__init__.py`，负责统一导出公共函数
- 子目录内的文件按职责拆分，文件名体现具体职责
- 不得在 `app/services/` 下出现同一模块既存在单文件又存在子目录的情况
- Constants 层的对应规则不变：无论 service 是单文件还是子目录，constants 文件名与 service 模块名一致

---

### 2.2 通用文件说明

项目包含一系列通用文件，详细说明请参考：[查看通用文件详细说明](./通用文件说明.md)

### 2.3 代码依赖方向

只允许以下依赖关系：

```text
API -> Services
API -> Schemas
Services -> Models
Services -> Schemas
Services -> Constants
Models -> Database
```

数据流转方向：

```text
Client -> API -> Schemas -> Services -> Models -> Database
Database -> Models -> Services -> Schemas -> API -> Client
```

规则：

- API 层只能调用 Services，不能直接操作 Models
- API 层可以依赖 Schemas 用于入参/出参声明
- Services 层可以依赖 Models 和 Schemas
- Models 层不得依赖 Services、API、Schemas
- 不允许反向依赖
- 不允许跨层混用职责

---

### 2.4 强制规则

- API 层只能做协议接入、依赖注入、调用服务、包装响应
- Services 层只能做业务逻辑、异常转换、事务提交
- Schemas 层只能做校验、序列化、结构定义
- Models 层只能做数据结构定义
- Database 细节不得泄漏到 API 层

---

### 2.5 禁止项

- 禁止 API 层直接写核心业务逻辑
- 禁止 API 层直接查询数据库模型
- 禁止在 Schema 中写业务规则
- 禁止在 Model 中写接口返回逻辑
- 禁止绕过 Services 直接拼响应数据
- 禁止 Services 中定义路由
- 禁止同一项目同时存在“函数式 Services”和“类 Services”两套体系

---

### 2.6 数据库会话职责边界

`get_db` 只负责数据库会话生命周期管理，不承担事务提交职责。

标准职责：

- `get_db`：创建会话 → `yield` 会话 → 异常兜底 `rollback()` → `close()`
- Services：执行读写、控制事务、调用 `commit()/rollback()/refresh()`

标准写法：

```python
def get_db():
    db = SessionLocal()
    try:
        yield db
    except Exception:
        db.rollback()
        raise
    finally:
        db.close()
```

规则：

- `get_db` 不得隐式 `commit()`
- `get_db` 不在 `finally` 中提交事务
- 写操作由 Services 层负责提交
- 兜底回滚可以保留，但不替代 Services 的显式异常处理

## 3. 命名规则

### 3.1 Models

- 表名：单数、下划线命名例：`resume`、`job_position`
- 类名：PascalCase例：`Resume`、`JobPosition`
- 字段名：全小写下划线
  例：`candidate_name`、`created_at`

---

### 3.2 Schemas

#### 命名规则

请求类统一使用 `*Request`，响应类统一使用 `*Response`。

允许的命名形式：

- 请求类：`XxxRequest`（涵盖所有输入场景，包括创建、更新、查询、筛选等）
- 详情响应：`XxxResponse` / `XxxDetailResponse`
- 列表项响应：`XxxItemResponse` / `XxxListItem`
- 枚举类：`XxxStatus`、`XxxType`、`XxxEnum`
- 通用类：`ApiResponse`、`PaginationInfo`、`PaginatedResponse`

规则：

- 基础模式：统一使用 `*Request` 和 `*Response`，不强制拆分
- 列表查询：统一使用 `{Entity}ListRequest`
- 完整 CRUD 场景：可按需拆分 `*CreateRequest` / `*UpdateRequest`
- 不得在同一 Schema 文件中混用 `*Request` 与 `*Create/*Update/*Query` 两套命名
- 响应类统一使用 `*Response`

---

### 3.3 Services

- 使用模块函数形式
- 文件名：业务名小写，如 `resume.py`、`user.py`
- 方法名：动词短语，如 `upload_resume`、`query_resume_list`
- 不生成 `XxxService` 类

#### CRUD 函数命名规范

| 操作     | 命名风格                                          | 示例                                              |
| -------- | ------------------------------------------------- | ------------------------------------------------- |
| 查询单个 | `get_{entity}_by_id` / `get_{entity}_detail`  | `get_position_by_id`, `get_recordings_detail` |
| 查询列表 | `get_{entity}_list` / `query_{entity}_list`   | `get_position_list`, `query_resume_list`      |
| 创建     | `create_{entity}` / `save_{entity}`           | `create_position`, `save_question`            |
| 更新     | `update_{entity}` / `update_{entity}_{field}` | `update_position`, `update_position_status`   |
| 删除     | `delete_{entity}`                               | `delete_position`, `delete_question`          |

规则：

- 函数名必须体现业务动作
- 查询单个必须包含 `by_id` 或 `detail` 后缀
- 查询列表使用 `list` 后缀
- 删除操作使用 `delete_` 前缀

---

### 3.4 API

- 路由文件名：业务名小写，如 `resume.py`、`auth.py`
- 路由变量统一为 `router`
- 方法名使用业务语义名，不强制 `get_` / `post_` 前缀

---

## 4. Models 规范

### 4.1 基础规则

每个模型必须满足：

- 主键字段为 `id`
- 主键类型使用 `BigInteger`
- 主键必须 `primary_key=True`
- 主键必须 `autoincrement=True`
- 所有字段必须写 `comment`
- 公共时间字段必须包含 `created_at`、`updated_at`
- 如使用软删除，字段名固定为 `is_deleted`

---

### 4.2 推荐类型映射

| SQL        | SQLAlchemy   |
| ---------- | ------------ |
| VARCHAR(n) | String(n)    |
| TEXT       | Text()       |
| INT        | Integer      |
| BIGINT     | BigInteger   |
| TINYINT    | SmallInteger |
| JSON       | JSON         |
| DATETIME   | DateTime     |

---

### 4.3 索引规则

下列字段应加 `index=True`：

- 外键字段
- 高频查询字段
- 状态字段

---

### 4.4 时间字段规则

- `created_at` 使用 `server_default=func.now()`
- `updated_at` 使用 `server_default=func.now()` 和 `onupdate=func.now()`
- 不使用 Python 层默认时间替代数据库默认值

---

### 4.5 状态字段规则

状态字段必须满足：

- 类型为 `SmallInteger`
- 必须有默认值
- 必须建立索引
- `comment` 中必须写清枚举含义

---

### 4.6 软删除规则

如使用软删除：

- 字段名固定：`is_deleted`
- 类型固定：`SmallInteger`
- 默认值固定：`0`
- 语义固定：`0=正常，1=已删除`

---

### 4.7 删除操作规范

删除操作分为软删除和硬删除两种，项目默认使用软删除。

#### 4.7.1 软删除

适用于有数据关联、需保留历史记录的业务实体：

```python
def delete_position(db: Session, position_id: int) -> dict:
    """删除岗位（软删除）"""
    stmt = select(JobPosition).where(
        and_(
            JobPosition.id == position_id,
            JobPosition.is_deleted == 0,
        )
    )
    db_position = db.scalar(stmt)
    if not db_position:
        raise ServiceException(ErrorCode.DATA_NOT_FOUND, "岗位不存在")

    db_position.is_deleted = 1
    try:
        db.commit()
    except Exception:
        db.rollback()
        raise ServiceException(ErrorCode.INTERNAL_ERROR, "操作失败")

    return {"id": position_id}
```

规则：

- 查询时必须过滤 `is_deleted == 0`
- 更新时设置 `is_deleted = 1`，不执行 `db.delete()`
- 软删除实体不物理移除数据

#### 4.7.2 硬删除

适用于无需保留、可彻底清除的数据：

```python
def delete_question(db: Session, question_id: int) -> bool:
    """删除题目（硬删除）"""
    stmt = select(InterviewQuestion).where(
        InterviewQuestion.id == question_id
    )
    question_entity = db.scalar(stmt)
    if not question_entity:
        raise ServiceException(ErrorCode.DATA_NOT_FOUND, "题目不存在")

    try:
        db.delete(question_entity)
        db.commit()
    except Exception:
        db.rollback()
        raise ServiceException(ErrorCode.INTERNAL_ERROR, "操作失败")

    return True
```

规则：

- 硬删除直接调用 `db.delete(entity)`
- 删除前必须确认实体存在
- 删除后不可恢复

#### 4.7.3 删除操作选择原则

| 场景                                 | 推荐方式   | 说明                       |
| ------------------------------------ | ---------- | -------------------------- |
| 岗位、简历、用户等核心业务实体       | 软删除     | 保留历史数据，支持数据恢复 |
| 面试题、录音、摘要等可重新生成的数据 | 硬删除     | 无需保留，节省存储空间     |
| 有外键关联的数据                     | 优先软删除 | 避免关联数据孤岛           |

---

### 4.8 CRUD 操作方法规范

项目使用 SQLAlchemy 2.0 风格，统一使用 `select()` 语句。

#### 4.8.1 查询单个

```python
# 方式一：使用 select() + db.scalar()（推荐）
def get_position_by_id(db: Session, position_id: int):
    stmt = select(JobPosition).where(
        and_(
            JobPosition.id == position_id,
            JobPosition.is_deleted == 0,
        )
    )
    db_position = db.scalar(stmt)
    if not db_position:
        raise ServiceException(ErrorCode.DATA_NOT_FOUND, "岗位不存在")
    return db_position

# 方式二：使用 db.get()（仅用于无软删除场景）
def get_recordings_detail(db: Session, recording_id: int):
    recording = db.get(InterviewRecording, recording_id)
    if not recording:
        raise ServiceException(ErrorCode.DATA_NOT_FOUND, "录音不存在")
    return recording
```

规则：

- 有软删除的实体必须使用 `select() + where(is_deleted == 0)` 模式
- 无软删除的实体可使用 `db.get()`
- 查询结果必须检查是否为 `None`

#### 4.8.2 创建

```python
def create_position(db: Session, data: PositionCreateRequest):
    db_position = JobPosition(**data.model_dump())
    db.add(db_position)
    try:
        db.commit()
    except Exception:
        db.rollback()
        raise ServiceException(ErrorCode.INTERNAL_ERROR, "操作失败")
    db.refresh(db_position)
    return db_position
```

规则：

- 使用 `model_dump()` 将 Request 转为字典
- 使用 `**kwargs` 解包构造实体
- `db.refresh()` 必须放在 `commit()` 成功之后

#### 4.8.3 更新

```python
# 方式一：动态更新（推荐，用于 UpdateRequest）
def update_position(db: Session, position_id: int, data: PositionUpdateRequest):
    stmt = select(JobPosition).where(
        and_(
            JobPosition.id == position_id,
            JobPosition.is_deleted == 0,
        )
    )
    db_position = db.scalar(stmt)
    if not db_position:
        raise ServiceException(ErrorCode.DATA_NOT_FOUND, "岗位不存在")

    update_data = data.model_dump(exclude_unset=True)
    for key, value in update_data.items():
        setattr(db_position, key, value)

    try:
        db.commit()
    except Exception:
        db.rollback()
        raise ServiceException(ErrorCode.INTERNAL_ERROR, "操作失败")

    db.refresh(db_position)
    return db_position

# 方式二：直接赋值（用于更新单个或少量字段）
def update_resume_status(db: Session, resume_id: int, status: int):
    db_resume = db.get(Resume, resume_id)
    if not db_resume:
        raise ServiceException(ErrorCode.DATA_NOT_FOUND, "简历不存在")

    db_resume.status = status
    try:
        db.commit()
    except Exception:
        db.rollback()
        raise ServiceException(ErrorCode.INTERNAL_ERROR, "操作失败")

    return {"id": resume_id, "status": status}
```

规则：

- 使用 `model_dump(exclude_unset=True)` 只更新传入的字段
- 使用 `setattr()` 动态设置属性
- `db.refresh()` 必须放在 `commit()` 之后

#### 4.8.4 批量创建

```python
def save_questions(db: Session, questions: list[QuestionItem]):
    saved_entities = []
    for item in questions:
        db_question = InterviewQuestion(**item.model_dump(), is_saved=1)
        db.add(db_question)
        saved_entities.append(db_question)

    db.flush()  # 获取 ID 但不提交
    try:
        db.commit()
    except Exception:
        db.rollback()
        raise ServiceException(ErrorCode.INTERNAL_ERROR, "操作失败")

    return saved_entities
```

规则：

- 批量操作使用 `db.flush()` 获取 ID
- 最终统一 `commit()`
- 失败时统一 `rollback()`

## 5. Schemas 规范

### 5.1 基础规则

- 所有 Schema 必须继承 `BaseModel`
- 请求类用于入参校验
- 响应类用于返回数据结构
- 通用响应必须使用统一响应包装
- 响应类必须开启 `from_attributes=True`

---

### 5.2 文件布局规则

Schema 文件中的内容按以下顺序排列：

1. 辅助类
3. 请求类
4. 响应类

即：

```text
辅助类 -> 请求类 -> 响应类
```

辅助类包括：

- `Enum`
- 被请求类/响应类引用的内部对象类

每个区域使用注释分隔符明确标识，分隔符统一使用以下格式：

```python
# ========== 辅助类（Support）==========  # 内部 Enum、通用对象类

# ========== 请求类（Request）==========  # 入参校验

# ========== 响应类（Response）==========  # 返回数据结构
```

规则：

- 分隔注释必须覆盖四个区域，不得省略或合并
- 不得在同一区域混入其他类型定义
- 注释风格统一为 `# ========== 区域名（英文名）==========`，风格需在项目内保持一致

---

### 5.3 请求类规则

- 必填字段使用 `Field(...)`
- 可选字段使用 `X | None = Field(None)`
- 字符串按需限制 `min_length`、`max_length`
- 数值按需限制 `ge`、`gt`、`le`、`lt`
- 字段尽量写 `description`
- `validator`/`field_validator` 仅用于字段值合法性校验，不得编写业务规则逻辑
- 不得在请求类中根据传入字段计算或派生其他字段值

---

### 5.4 响应类规则

- 响应类必须开启 `from_attributes=True`
- 响应类声明返回字段，字段值的生成和赋值由 services 层负责
- 不得在响应类中使用业务逻辑推导字段值

---

### 5.5 枚举规则

- 枚举优先使用 `IntEnum`
- 枚举值必须与数据库状态值一致
- 枚举成员使用大写下划线命名

---

### 5.6 通用响应原语

列表接口返回：

```python
ApiResponse[PaginatedResponse[T]]
```

详情接口返回：

```python
ApiResponse[XxxResponse]
```

分页响应构造方式：

```python
# Services 层：构造 PaginatedResponse 并返回
from app.schemas.common import PaginatedResponse, PaginationInfo

return PaginatedResponse(
    lists=response_list,  # list[XxxItemResponse]
    pagination=PaginationInfo(
        total=total,
        total_pages=math.ceil(total / page_size),
        page=page,
        page_size=page_size,
    ),
)

# API 层：声明 response_model，返回时用 success() 包装
@router.get("", response_model=ApiResponse[PaginatedResponse])
def query_list(query: XxxRequest = Depends(), db: Session = Depends(get_db)):
    result = services.query_xxx_list(query, db)
    return success(data=result)  # result 已是 PaginatedResponse
```

规则：

- 分页列表数据结构只能使用 `PaginatedResponse`
- 不允许另起一套 `ListResult`、`PageResult`、`PageVO` 等同义包装
- Services 层返回 `PaginatedResponse` 实例，不返回 `ApiResponse`
- API 层用 `success(data=result)` 包装 `PaginatedResponse`
- 具体构造动作由 `success()` / `error()` 完成

### 5.7 职责边界规则

Schemas 层只承担以下职责：

1. **数据结构定义**：声明字段名称与类型
2. **类型约束**：使用 Pydantic Field 定义必填、可选、范围约束
3. **字段校验**：使用 `validator`/`field_validator` 做单字段合法性子句
4. **序列化 / 反序列化**：Pydantic 原生序列化行为控制
5. **输入输出边界描述**：明确接口契约中"进什么、出什么"

Schemas 层不承担以下职责：

| 序号 | 排除事项       | 说明                                     |
| ---- | -------------- | ---------------------------------------- |
| 1    | 业务逻辑       | 不写 if/else、分支判断等业务规则         |
| 2    | 状态流转       | 不负责状态的推导与变换                   |
| 3    | 展示字段推导   | 不推导前端展示所需的派生字段值           |
| 4    | 衍生字段赋值   | 不计算、拼接、格式化业务含义的字段内容   |
| 5    | 默认业务值填充 | 不填充需要查询或计算才能得到的业务默认值 |
| 6    | 跨字段业务决策 | 不根据多个字段联动产生业务判断           |
| 7    | 结果组装       | 不组装跨实体的聚合数据                   |

以上业务层职责统一由 services 层处理。Response Schema 声明返回字段，字段值的生成和赋值属于 services 层职责。

---

## 6. Services 规范

### 6.1 基础规则

- Services 使用模块函数实现
- 函数名必须体现业务动作
- Services 负责核心业务逻辑
- Services 负责业务异常转换
- Services 不负责 HTTP 协议层处理
- **函数不使用返回值类型注解**（不写 `-> XxxResponse`）
- **每个函数必须书写函数级 docstring，一行简述功能即可，不写参数与返回值说明**
- **每个文件必须在首部添加文件级 docstring，一行简述文件功能**

---

### 6.2 文件首部说明规则

每个文件必须在首部（import 语句之前）添加文件级 docstring，一行简述文件功能。

```python
"""财报记录删除服务"""
from sqlalchemy import select
...
```

规则：

- 文件级 docstring 必须位于文件首行，在所有 import 语句之前
- 使用三引号 `"""` 包裹
- 内容为一句简短的功能描述，说明该文件负责什么业务
- 不需要详细描述参数、返回值等（那是函数级 docstring 的职责）
- `__init__.py` 文件同样需要添加，描述该模块的导出职责

---

### 6.3 文件布局规则

每个 Service 文件按以下顺序组织：

1. 公共入口函数（被 API 层直接调用的函数）
2. `"""辅助函数"""` 分隔注释
3. 仅被本文件内部调用的私有辅助函数

```python
# ========== 公共入口函数 ==========

def query_candidate_list(query: CandidateQueryRequest, db: Session):
    """查询候选人列表"""

def query_candidate_detail(candidate_id: int, db: Session):
    """查询候选人详情"""

def create_candidate(data: CandidateCreateRequest, db: Session):
    """创建候选人"""

"""辅助函数"""


def _build_candidate_response(entity: Candidate):
    """构建候选人响应结构"""
```

规则：

- 公共入口函数放在文件上部，供 API 层直接调用
- 辅助函数放在分隔注释之后，仅限本文件内部使用
- 分隔注释统一使用 `"""辅助函数"""`
- 辅助函数以单下划线 `_` 开头命名（如 `_xxx`）

---

### 6.4 输入输出唯一规则

输入优先接收：

- Request Schema
- 基础类型
- 数据库会话 `db`

输出**只能返回业务数据对象**，例如：

- `XxxResponse`
- `PaginatedResponse[XxxItemResponse]`
- 少量简单基础类型（仅当业务明确需要）

禁止返回：

- `ApiResponse`
- `success()/error()` 的结果
- ORM 实体对象
- “统一错误结果”
- `None` 作为失败信号

规则：

- Service 成功只返回正常业务数据
- Service 失败只通过 `raise ServiceException(...)`
- Service 不负责包装最外层响应壳
- API 负责把 Service 返回的数据包装为 `success(...)`

---

### 6.5 异常规则

- 文件校验失败：`3001/3002`
- 数据不存在：`1002`
- 参数不合法：`1001`
- 外部服务失败：`4001`
- 未预期异常：`5001`

代码中统一写法：

```python
raise ServiceException(ErrorCode.DATA_NOT_FOUND, "简历不存在")
```

禁止：

```python
raise ServiceException(1002, "简历不存在")
```

---

### 6.6 数据库事务规则

#### 6.6.1 提交职责

谁执行写操作，谁负责本地 `commit()`。

规则：

- 查询操作不强制 `rollback()`
- 写操作必须在同一处 `try-except` 中完成 `commit()`
- `db.refresh()` 必须放在 `commit()` 成功之后
- 不允许在 `get_db` 中隐式 `commit()`

---

#### 6.6.2 提交失败处理

`commit()` 失败后必须立即 `rollback()`，然后转换为 `ServiceException`。

规则：

- 每个 `db.commit()` 必须显式配对 `db.rollback()`
- `commit()` 失败不允许直接 re-raise 原始异常
- 失败后必须立即 `db.rollback()`
- 统一转换为 `ServiceException(ErrorCode.INTERNAL_ERROR, "操作失败")`

---

#### 6.6.3 默认风格 + 公共事务辅助函数

`commit_or_rollback` 作为公共事务辅助函数，统一放置在 `app/db/database.py`。

推荐模板：

```python
from app.db.database import commit_or_rollback
from app.schemas.common import ErrorCode
from app.utils.exception import ServiceException

def update_resume(resume_id: int, data: ResumeUpdateRequest, db: Session) -> ResumeResponse:
    entity = db.get(Resume, resume_id)

    if not entity:
        raise ServiceException(ErrorCode.DATA_NOT_FOUND, "简历不存在")

    for key, value in data.model_dump(exclude_unset=True).items():
        setattr(entity, key, value)

    commit_or_rollback(db)

    db.refresh(entity)
    return ResumeResponse.model_validate(entity)
```

`commit_or_rollback` 封装了 `try-except + rollback + ServiceException` 的标准模式，Services 层直接调用即可。

查询模板：

```python
from app.schemas.common import ErrorCode
from app.utils.exception import ServiceException

def query_resume_detail(resume_id: int, db: Session) -> ResumeResponse:
    entity = db.get(Resume, resume_id)

    if not entity:
        raise ServiceException(ErrorCode.DATA_NOT_FOUND, "简历不存在")

    return ResumeResponse.model_validate(entity)
```

### 6.7 分页查询规范

分页查询使用 SQLAlchemy 2.0 风格的 `select()` + `func.count()` + `offset/limit`。

标准实现：

```python
from sqlalchemy import select, func
import math

def get_position_list(db: Session, query: PositionListRequest):
    # 1. 构建基础查询（包含软删除过滤）
    base_stmt = select(JobPosition).where(JobPosition.is_deleted == 0)

    # 2. 动态添加筛选条件
    if query.position_name:
        base_stmt = base_stmt.where(
            JobPosition.position_name.like(f"%{query.position_name}%")
        )
    if query.department:
        base_stmt = base_stmt.where(JobPosition.department == query.department)

    # 3. 计算总数
    total = db.scalar(select(func.count()).select_from(base_stmt.subquery()))

    # 4. 执行分页查询
    items = (
        db.scalars(
            base_stmt
            .order_by(JobPosition.created_at.desc())
            .offset((query.page - 1) * query.page_size)
            .limit(query.page_size)
        ).all()
    )

    # 5. 返回 PaginatedResponse
    return PaginatedResponse(
        lists=items,
        pagination=PaginationInfo(
            page=query.page,
            page_size=query.page_size,
            total=total,
            total_pages=math.ceil(total / query.page_size) if total else 0,
        ),
    )
```

规则：

- 使用 `func.count()` 获取总数
- 使用 `offset/limit` 实现分页
- `total_pages` 使用 `math.ceil(total / page_size)` 计算
- `total` 为 0 时 `total_pages` 返回 0
- 查询列表必须按 `created_at.desc()` 倒序排列

---

### 6.8 Schema 与实体转换规范

ORM 实体与 Pydantic Schema 之间的转换使用 `model_validate()` 和 `model_dump()`。

#### 6.8.1 实体转响应（ORM -> Schema）

```python
# 单个实体
response = PositionResponse.model_validate(db_position)
return response

# 列表转换
lists = [PositionResponse.model_validate(item) for item in items]
return PaginatedResponse(lists=lists, pagination=...)
```

规则：

- 统一使用 `model_validate(entity)` 方法
- 不使用 `PositionResponse.from_orm(entity)`（已废弃）

#### 6.8.2 请求转字典（Schema -> dict）

```python
# 全部字段
data = request.model_dump()

# 仅更新传入的字段（exclude_unset=True）
update_data = request.model_dump(exclude_unset=True)
```

规则：

- 创建时使用 `model_dump()` 获取全部字段
- 更新时使用 `model_dump(exclude_unset=True)` 仅获取传入字段
- `exclude_unset=True` 会排除未显式设置的字段（包括有默认值的）

#### 6.8.3 字典转实体（dict -> ORM）

```python
# 方式一：解包构造（推荐）
db_entity = JobPosition(**data.model_dump())

# 方式二：逐字段赋值
for key, value in update_data.items():
    setattr(db_entity, key, value)
```

规则：

- 创建时使用 `**kwargs` 解包构造实体
- 更新时使用 `setattr()` 逐字段赋值

#### 6.8.4 常见错误

禁止：

```python
# 错误：使用已废弃的 from_orm
response = PositionResponse.from_orm(db_position)

# 错误：使用 update() 方法（SQLAlchemy 1.x 风格）
db_position.update(data)  # 错误方式

# 错误：model_dump(exclude_none=True) 用于更新
update_data = request.model_dump(exclude_none=True)  # 会排除 None 值，导致无法设为空
```

## 7. API 规范

### 7.1 路由规则

- 路由前缀统一为 `/api/v1/xxx`
- 使用 `APIRouter`
- 路由对象名固定为 `router`

---

### 7.2 职责规则

API 层只做以下事情：

- 接收请求参数
- 注入依赖
- 调用 Services
- 使用 `success()` / `error()` 包装统一响应
- 处理认证依赖

---

### 7.3 唯一返回方式

API 层成功返回：

```python
return success(data=result)
```

API 层失败返回：

```python
return error(code=e.code, message=e.message)
```

推荐写法：

```python
from app.schemas.response import success, error
from app.utils.exception import ServiceException
from app.schemas.common import ApiResponse, PaginatedResponse

@router.get("/{id}", response_model=ApiResponse[ResumeResponse])
def query_resume_detail(id: int, db: Session = Depends(get_db)):
    try:
        result = services_resume.query_resume_detail(id, db)
        return success(data=result)
    except ServiceException as e:
        return error(code=e.code, message=e.message)
```

列表接口示例：

```python
@router.get("", response_model=ApiResponse[PaginatedResponse[ResumeItemResponse]])
def query_resume_list(query: ResumeQueryRequest = Depends(), db: Session = Depends(get_db)):
    try:
        result = services_resume.query_resume_list(query, db)
        return success(data=result)
    except ServiceException as e:
        return error(code=e.code, message=e.message)
```

规则：

- API 层不得直接返回 ORM 对象
- API 层不得直接返回 Service 原始异常
- API 层不得直接拼接非统一格式 JSON
- API 层不得再定义第二套 success/error helper

---

### 7.4 依赖注入规则

默认通过 `Depends()` 注入：

- `get_db`
- `get_current_user`

## 8. 认证授权规范

### 8.1 认证方式

系统统一使用 JWT Bearer Token。

固定约束：

- 协议：OAuth2 Password Bearer
- Token 类型：Bearer
- Token 格式：JWT
- 客户端通过 `Authorization: Bearer <token>` 传递 Token

---

### 8.2 Token 规则

- 登录成功后返回 `access_token`
- Token 载荷中至少包含用户标识
- Token 过期必须返回 `2002`
- 无 Token 或未认证返回 `2001`
- 权限不足返回 `2003`
- 账号密码错误返回 `6001`

---

### 8.3 当前用户依赖

AI 生成受保护接口时，默认使用：

- `get_current_user` 解析当前登录用户
- `get_db` 获取数据库会话

---

### 8.4 登录接口规则

登录接口必须：

- 校验用户名和密码
- 凭证正确时生成 JWT
- 凭证错误时返回 `6001`
- 返回统一响应结构

---

### 8.5 登出接口规则

登出默认由客户端删除本地 Token。
服务端可返回成功响应，不强制维护服务端会话状态。

## 9. 异常处理规范

### 9.1 总原则

- 统一返回格式
- 面向用户的错误信息必须简洁
- 不暴露内部实现、堆栈、文件路径
- 详细异常仅用于日志
- 未分类异常向上转换为统一错误响应

---

### 9.2 分层处理原则

- API 层：接收参数、调用服务、返回统一结构
- Services 层：处理业务异常并映射错误码
- Models 层：不处理业务异常

---

### 9.3 用户可见信息规则

用户可见错误信息必须：

- 简短
- 可理解
- 无内部实现细节
- 尽量可操作

允许示例：

- `"参数错误"`
- `"简历不存在"`
- `"不支持的文件格式"`
- `"服务调用失败，请稍后重试"`
- `"操作失败"`
- `"系统内部错误"`

禁止输出：

- 文件路径
- Python 异常堆栈
- 第三方服务原始报错
- SQL 原文
- 内部模块路径
- 数据库驱动错误详情

---

### 9.4 ServiceException 异常类

统一定义：

```python
class ServiceException(Exception):
    def __init__(self, code: int, message: str):
        self.code = code
        self.message = message
        super().__init__(message)
```

---

### 9.5 ServiceException 唯一使用方式

Services 层使用：

```python
raise ServiceException(ErrorCode.DATA_NOT_FOUND, "简历不存在")
```

规则：

- Services 层用 `raise ServiceException(...)` 替代 `return None`
- API 层用 `try-except` 统一捕获并返回 `error(...)`
- 错误码必须使用 `ErrorCode` 枚举
- 不允许硬编码数字错误码

---

### 9.6 参数校验唯一规则

参数校验分两层：

1. **结构校验**：由 FastAPI + Pydantic Request Schema 负责
2. **业务前置校验**：由 Services 中 `if` 判断后抛 `ServiceException`

示例：

```python
if page_size <= 0:
    raise ServiceException(ErrorCode.PARAM_ERROR, "分页参数错误")
```

规则：

- 请求体格式、字段类型、基础约束交给 Schema
- 业务前置条件在 Service 层判断
- 不允许把业务规则塞进 Schema
- 不允许用 `return None` 表示参数错误

---

### 9.7 第三方服务异常

调用 LLM、Milvus、外部 API 等第三方服务时：

- 日志中可记录详细异常
- 返回给用户的错误必须脱敏
- 统一映射到 `ErrorCode.AI_SERVICE_ERROR`

推荐写法：

```python
try:
    result = client.call(...)
except Exception:
    raise ServiceException(ErrorCode.AI_SERVICE_ERROR, "服务调用失败，请稍后重试")
```

规则：

- 第三方服务失败不使用 `5001`
- 不向用户暴露原始第三方报错
- 统一在 Services 层完成异常转换

---

### 9.8 数据库异常处理模式

数据库异常与业务异常遵循分层处理原则：

- 业务层异常：`ServiceException + 业务错误码`
- 数据库提交失败：`rollback()` 后转换为 `ServiceException(ErrorCode.INTERNAL_ERROR, "操作失败")`

推荐模板：

```python
try:
    db.commit()
except Exception:
    db.rollback()
    raise ServiceException(ErrorCode.INTERNAL_ERROR, "操作失败")
```

规则：

- `commit()` 失败后必须立刻 `rollback()`
- 不允许直接抛出原始数据库异常
- 用户可见信息统一为 `"操作失败"` 或 `"系统内部错误"`

## 10. 配置规范

### 10.1 配置来源

配置分为三类：

- `.env`：敏感配置与环境变量
- `app/core/config.py`：统一配置入口
- `app/config/*.yaml`：业务模板、Prompt、规则配置

---

### 10.2 Settings 规则

- 所有配置统一由 `Settings` 管理
- `Settings` 必须继承 `BaseSettings`
- 所有字段必须有类型注解
- 敏感配置优先从环境变量读取
- 派生配置使用 `@property`
- 全局只使用单例 `settings`

---

### 10.3 访问规则

- 业务代码统一 `from app.core.config import settings`
- 不允许在业务代码中直接读取环境变量
- YAML 业务配置通过统一配置对象访问
- 不允许在多个地方重复定义相同配置项

---

### 10.4 环境变量规则

- 使用全大写下划线命名
- 敏感信息不得硬编码
- 非敏感配置可提供默认值
- 生产环境必须替换默认密钥和默认密码

---

### 10.5 配置禁止项

- 禁止硬编码 `SECRET_KEY`
- 禁止在业务代码中直接 `os.getenv()` 读取核心配置
- 禁止多套配置入口并存
- 禁止在代码中散落数据库连接串

---

## 11. 模块导出规则

- 各层可在 `__init__.py` 中集中导出核心对象
- 导出必须与当前项目真实实现一致
- 不得同时维护”类服务”和”函数服务”两套导出规范
- 若项目采用模块函数式 Services，则不要生成 `XxxService` 类导出

---

## 11.1 模块导入命名规范

### 别名导入推荐写法

当同一模块名在 schemas、services、models 多层存在时，推荐使用别名导入明确来源：

```python
from app.schemas import questions as schemas_questions
from app.services import questions as services_questions
from app.models import questions as models_questions

from app.models import position as models_position
from app.models import resume as models_resume
```

### 别名命名规则

| 源模块       | 推荐别名     | 示例                   |
| ------------ | ------------ | ---------------------- |
| schemas/xxx  | schemas_xxx  | `schemas_questions`  |
| services/xxx | services_xxx | `services_questions` |
| models/xxx   | models_xxx   | `models_position`    |

### 使用场景

- 同一模块名跨多层存在时（如 `questions` 同时存在于 schemas/services/models）
- 提升代码可读性，明确标识数据来源
- 避免命名歧义（如 `QuestionType` vs `models_questions.QuestionType`）

### 注意事项

- 别名应统一风格：`层名_模块名`
- 避免使用无意义别名如 `q`、`qs`
- 与直接导入不冲突，公共模块仍可直接导入

## 12. AI 生成代码默认优先级

当规范冲突时，按以下优先级执行：

1. 本文硬规则
2. 当前项目已有目录与已有实现风格
3. 通用框架最佳实践
4. 示例代码

规则：

- 示例不是第一优先级
- 若示例与硬规则冲突，以硬规则为准
- 若历史代码与硬规则冲突，优先遵守硬规则并保持局部兼容

---

## 13. AI 禁止生成的模式

AI 生成代码时，禁止出现以下模式：

- API 层直接操作数据库模型
- Schemas 中编写业务逻辑
- Services 中直接处理路由声明
- 返回结构不包含 `code/message/data`
- 列表接口不包含 `pagination`
- 直接暴露原始异常
- 硬编码敏感信息
- 使用未定义错误码段
- 认证接口绕过 JWT Bearer 规范
- 同一项目同时生成互相冲突的导入路径或实现方式
- 同一 Schema 文件中混用 `*Request` 与 `*Create/*Update/*Query` 两套命名
- Services 同时返回 DTO、ORM、错误对象三套返回风格
- API 层同时使用 `success()/error()` 与手写 JSON 两套响应风格

---

## 14. 最小实现模板

AI 生成接口时默认遵循以下最小流程：

```text
Request
-> API 接收参数与依赖
-> Schema 校验
-> Service 执行业务
-> Model 读写数据
-> Service 产出业务数据对象
-> API 使用 success(...) 返回
    -> 若捕获 ServiceException：返回 error(code, message)
```

规则：

- `get_db` 注入会话，由 Services 持有和使用
- Services 负责事务提交与回滚，不在 `get_db` 中隐式提交
- API 层不直接操作数据库
- API 层只负责接收、调用、包装返回

---

## 15. 唯一方案总结

本项目唯一允许的默认实现方式如下：

- **Schema 命名**：统一 `*Request/*Response`
- **Services 形态**：统一模块函数
- **Service 成功返回**：统一返回业务数据对象
- **Service 失败返回**：统一 `raise ServiceException(...)`
- **API 成功返回**：统一 `return success(data=...)`
- **API 失败返回**：统一 `return error(code=e.code, message=e.message)`
- **错误码使用**：统一 `ErrorCode.xxx`
- **数据库事务**：统一由 Services 负责 `commit()/rollback()`
- **统一响应壳**：统一符合 `ApiResponse`
- **分页结构**：统一 `PaginatedResponse`，Services 层构造 `PaginatedResponse` 实例返回，API 层用 `success(data=result)` 包装

## 16. Constants 规范

### 16.1 基础规则

- Constants 层与 services 层目录结构一一对应
- 只有含有常量的 service 才创建对应的 constants 模块
- 文件命名与对应的 service 文件名一致
- 每个 constants 文件必须在首行添加文件级 docstring

### 16.2 文件首部说明规则

每个文件必须在首部（import 语句之前）添加文件级 docstring。

```python
"""Chat 服务常量定义"""
from app.models import ...
...
```

规则：

- 文件级 docstring 必须位于文件首行，在所有 import 语句之前
- 内容为一句简短的功能描述，说明该文件负责什么常量
- 使用三引号 `"""` 包裹

### 16.3 Constants 使用规则

Services 层通过 import 使用 constants：

```python
from app.constants import chat as constants_chat

# 使用常量
MAX_CONCURRENT = constants_chat.MAX_CONCURRENT
```

规则：

- 推荐使用别名导入，避免与本地变量名冲突
- Constants 模块不得依赖 services 层
